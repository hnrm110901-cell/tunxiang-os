# 屯象OS 微信支付对接技能

## 快速决策

先判断任务类型，再读取对应参考文件：

1. **接口开发/改造**（新增或修改支付 API、支付状态流转）
   - 读取 `references/tunxiang-wechat-pay-map.md`
2. **联调/验收**（沙箱联调、真机支付、回调通知）
   - 读取 `references/wechat-pay-go-live-checklist.md`
3. **故障排查**（支付成功但订单未完成、回调验签失败、重复扣款）
   - 两个参考文件都读

## 执行流程（必须按顺序）

1. **识别支付场景**
   - JSAPI（小程序）
   - Native 扫码
   - 付款码（被扫）
   - 退款
2. **确认当前实现层级**
   - 若是 `shared/integrations/wechat_pay.py` Mock 分支在生效，先补齐生产配置与验签。
   - 若 `services/tx-trade/src/api/wechat_pay_routes.py` 仍有 TODO（如回调落库），先补闭环再联调。
3. **落地改造**
   - 优先保证：幂等、签名验证、状态机一致性、失败补偿。
   - 所有写操作保留 `X-Tenant-ID` 租户隔离（微信回调例外）。
4. **联调与验收**
   - 使用 checklist 逐项验证：下单→支付→回调→订单状态→退款→查询。
5. **输出交付件**
   - 代码改造说明
   - 环境变量与证书配置清单
   - 风险项与回滚方案

## 屯象OS 对接硬约束

1. **禁止仅靠前端结果判定支付成功**，必须以后端回调或主动查单确认为准。
2. **支付/退款必须幂等**，重复请求不可重复扣款。
3. **回调验签失败必须拒绝并记录审计日志**。
4. **订单状态更新与支付流水写入要在同一业务事务语义下完成**（必要时通过 Saga/补偿保障）。
5. **不得在日志打印完整 openid、银行卡号、证书私钥内容**。

## 默认输出模板

当用户说“对接微信支付”但未给细节时，默认输出：

1. 当前代码现状（已实现/未实现）
2. 分阶段改造计划（P0 可用、P1 稳定、P2 合规）
3. 可直接执行的任务清单（文件级）
4. 联调用例（成功、失败、重复回调、超时补偿）
5. 上线检查与回滚预案
