# 屯象OS 产品自知识库

## 概述

本技能是屯象OS的**产品大脑**，存储经过验证的产品事实，用于快速回答产品相关问题。
避免每次都需要全量扫描代码库，提供基于事实的即时回答。

> **重要：** 本知识库的数据截止日期以 references 文件中标注的日期为准。
> 如果用户问到的功能可能在截止日期后有变化，应先扫描代码确认后再回答。

## 第一步：识别问题类型

| 问题类型 | 示例 | 参考文件 |
|---------|------|---------|
| **功能有无** | "有没有排班功能？" | `references/feature-matrix.md` |
| **技术架构** | "数据怎么同步的？" | `references/architecture-facts.md` |
| **竞品对比** | "和客如云比有什么优势？" | `references/competitive-edge.md` |
| **部署方案** | "一个门店需要什么硬件？" | `references/deployment-spec.md` |
| **客户案例** | "哪些客户在用？" | `references/customer-cases.md` |
| **产品边界** | "能不能做外卖？" | `references/feature-matrix.md` |
| **不确定** | → 先扫描代码再回答 | — |

## 第二步：读取参考

根据问题类型读取对应 reference 文件，然后回答。

**回答原则：**
- 有的功能说"有"，附带具体位置（服务名+文件）
- 没有的功能说"目前没有"，附带是否在开发计划中
- 部分完成的功能说"已有基础，需要XX补全"
- **不确定的不猜，去扫描代码**

## 第三步：回答框架

### 功能有无回答模板
```
✅ 有 — [功能名]
  位置：services/tx-xxx/src/...
  完成度：[完整/基础框架/部分实现]
  说明：[一句话描述当前能力]

❌ 没有 — [功能名]
  计划：[在开发计划Phase X / 未规划]
  替代：[如有临时替代方案]

⚠️ 部分 — [功能名]
  已有：[已完成的部分]
  缺失：[需要补全的部分]
  预估：[补全工作量]
```

### 竞品对比回答模板
```
| 维度 | 屯象OS | [竞品] | 差异分析 |
|------|--------|--------|---------|
| ... | ... | ... | 屯象优势/劣势/持平 |

核心差异化：
1. [屯象独有能力1]
2. [屯象独有能力2]

需要追赶：
1. [竞品领先点1]
```

## 核心铁律

1. **基于事实，不猜测** — 不确定就扫描代码，不编造功能
2. **标注完成度** — 不说"有"就完了，要说清楚完成到什么程度
3. **标注数据截止日期** — 每次回答注明知识库更新时间
4. **区分"代码存在"和"可用"** — 有代码不等于可交付，要考虑测试覆盖和集成状态
5. **竞品信息要客观** — 不贬低竞品，实事求是对比

## 知识库更新机制

当以下事件发生时，应更新 references 文件：
- 新功能开发完成并合并
- 新客户接入
- 架构重大变更
- 竞品有重大更新

更新方式：修改对应的 references/*.md 文件，标注更新日期。
