# /review — 全量审查流程

当用户输入 `/review` 时，对当前变更进行架构审查 + 代码审查 + 测试验证的全量审查。

可选参数：
- `/review` — 审查当前未提交的所有变更
- `/review <file_path>` — 审查指定文件
- `/review --staged` — 只审查已 stage 的变更

## 执行步骤

### Step 1: 收集变更

```bash
# 获取变更文件清单
git diff --name-only          # 未 stage 的变更
git diff --cached --name-only  # 已 stage 的变更
git diff HEAD                  # 所有未提交的变更
```

读取所有变更文件的 diff 内容。

### Step 2: 架构审查

按照 `.claude/agents/architect.md` 的审查维度执行：

- 分层合规性检查
- 多租户隔离检查
- 数据模型规范检查
- POS 集成合规检查
- 前端架构检查
- Agent 架构检查

### Step 3: 代码审查

按照 `.claude/agents/reviewer.md` 的审查维度执行：

- 安全性检查（SQL 注入、敏感数据泄露）
- 正确性检查（金额单位、UUID 类型匹配）
- 命名规范检查
- 产品规范检查（￥字段、决策型推送）
- 代码质量检查（死代码、重复实现）
- 前端专项检查（如涉及前端）

### Step 4: 测试验证

按照 `.claude/agents/tester.md` 的策略：

- 检查变更是否有对应测试
- 运行相关测试
- 报告覆盖盲区

### Step 5: 输出综合报告

```
# 全量审查报告

## 变更概述
- 变更文件数：X
- 新增行数：+XX / 删除行数：-XX
- 涉及模块：...

## 架构审查：[通过 / 有条件通过 / 拒绝]
（架构审查详情）

## 代码审查：[通过 / 需修改 / 拒绝]
（代码审查详情）

## 测试验证：[通过 / 部分通过 / 失败]
（测试详情）

## 总评
- BLOCKER 数量：X（必须修复）
- WARNING 数量：X（建议修复）
- NITPICK 数量：X（可选优化）

## 建议操作
1. [必须] ...
2. [建议] ...
3. [可选] ...
```

如有 BLOCKER 级别问题，明确告知不建议合并，并给出修复方案。
