# 屯象OS × 天财商龙SaaS — 商户替换能力开发规划（2026）

> **文档目标**：基于差距分析四份文档（gap-analysis-tiancailong-v7.7.86、gap-report-strategic-2026、v2-checklist-store-pos、v2-checklist-hq-reports），规划**使屯象OS具备完整替代天财商龙SaaS**的开发路线。  
> **策略原则**：不求1:1形态复刻，只求**业务结果等价 + 验收语言对齐**；屯象AI-Native差异化长板不可矮化。  
> **架构基准**：现有 v148 事件总线、16个微服务、10个前端App、边缘Mac mini、商米安卓POS。  
> **日期**：2026-04-04

## 二、分阶段开发规划

### Phase 1：P0刚需补全（2026 Q2，4月～6月）
**目标**：消除「明显缺失、客户必问」的验收死角，支撑中餐连锁直营客户签单。

#### 模块1.1：存酒管理系统 🔴→✅

**差距来源**：天财清单 `主菜单>存酒`（行业版+企业版），总部「营业管理>存酒」模块。  
**业务价值**：中餐/火锅大单店高频刚需，当前屯象完全空白。

**开发任务**：

**后端（tx-trade 或新增 tx-cellar 子路由）：**
- 数据模型：`wine_bottles`表（会员ID、桌台ID、品名、规格、数量、登记日期、有效期、状态）
- API：
  - `POST /api/v1/wine/register` — 存酒登记（关联会员/桌台）
  - `POST /api/v1/wine/withdraw` — 支取（数量、剩余更新）
  - `GET /api/v1/wine/query` — 查询（按会员/按桌台/按品名）
  - `POST /api/v1/wine/transfer` — 转桌带走存酒
  - `GET /api/v1/wine/ledger/{member_id}` — 存酒台账明细
  - `POST /api/v1/wine/write-off` — 报废/核销
- 事件接入：`emit_event(WineCellarEventType.DEPOSITED/WITHDRAWN/TRANSFERRED)`
- 打印：存酒单/支取单模板

**前端（web-pos）：**
- `WineStoragePage` — 存酒主界面（登记/支取/查询三标签）
- `WineRegisterModal` — 存酒登记弹窗（扫码/手输品名、关联客位）
- `WineWithdrawModal` — 支取弹窗（显示剩余量、输入支取量）
- `WineLedgerPage` — 存酒台账（按会员/按日期筛选，可打印）
- 主菜单「存酒」入口集成

**验收标准**：存酒登记→关联桌台→支取→打印支取单→会员台账查询完整闭环。

#### 模块1.2：餐桌押金专模 🟡→✅

**差距来源**：天财清单 `主菜单>押金`（5个子项），行业版+企业版均有。  
**现状**：`CreditPayPage`有零散雏形，无专用押金模型。

**开发任务**：

**后端（tx-trade）：**
- 数据模型：`table_deposits`表（桌台ID、金额、状态[冻结/退还/转消费]、操作员、时间戳）
- API：
  - `POST /api/v1/deposits/register` — 登记押金（关联客位）
  - `POST /api/v1/deposits/refund` — 退押金（全额/部分）
  - `POST /api/v1/deposits/convert` — 押金转消费（结账时抵扣）
  - `PUT /api/v1/deposits/{id}/associate` — 变更关联客位
  - `POST /api/v1/deposits/{id}/reprint` — 补打押金单
  - `GET /api/v1/deposits/query` — 当班押金台账
- 事件：`DEPOSIT.REGISTERED/REFUNDED/CONVERTED`

**前端（web-pos）：**
- `DepositPage` — 押金主界面（当班押金列表）
- `DepositRegisterModal` — 登记押金（客位选择+金额输入+备注）
- `DepositRefundModal` — 退押金（显示已登记金额）
- 结账流程集成：`CheckoutPage`中押金抵扣选项

**验收标准**：押金登记→关联客位→退押金/转消费→补打押金单完整流程，台账可查。

#### 模块1.3：协议单位与挂账闭环 🟡→✅

**差距来源**：天财总部功能清单「协议单位」Sheet（21项流程），天财POS「挂账」结算方式。

**开发任务**：

**后端（tx-org + tx-finance）：**
- 数据模型：
  - `agreement_units`（协议单位主数据：名称、联系人、额度、剩余额度、状态）
  - `agreement_signers`（签单人：姓名、工号、单位ID、权限额度）
  - `agreement_bills`（挂账单据：单位ID、签单人、金额、状态[未还/部分/已还]）
- API：
  - `POST /api/v1/agreement-units` — 新建协议单位
  - `GET /api/v1/agreement-units/{id}/limit` — 查询额度与余额
  - `POST /api/v1/agreement-bills/sign` — POS挂账操作
  - `POST /api/v1/agreement-bills/{id}/repay` — 还款（总部操作）
  - `GET /api/v1/agreement-units/{id}/statement` — 对账单导出
  - `POST /api/v1/agreement-units/{id}/disable` — 停用
- 事件：`AGREEMENT.SIGNED/REPAID/LIMIT_EXCEEDED`

**前端：**
- `web-admin`：协议单位CRUD、对账单管理、还款操作
- `web-pos`：结账时「挂账」支付方式 → 弹出签单人选择+额度验证

**验收标准**：总部建协议单位→门店POS挂账→签单人限额控制→总部还款对账完整闭环。

#### 模块1.4：最低消费/服务费规则引擎 🟡→✅

**差距来源**：天财POS支持最低消费补齐、账单级服务费，中餐正餐标配。

**开发任务**：

**后端（tx-trade规则引擎扩展）：**
- 配置模型：`billing_rules`（门店ID、规则类型[min_spend/service_fee]、计算方式、阈值、服务费率）
- 逻辑：
  - 结账时自动计算最低消费差额，提示补足方式（补充点菜/直接补差价）
  - 服务费按比例/固定金额计算，作为独立账单行项
  - 特定会员等级/协议单位免服务费配置
- API：
  - `GET /api/v1/billing-rules/{store_id}` — 获取门店规则
  - `PUT /api/v1/billing-rules/{store_id}` — 配置规则（管理员）
  - `POST /api/v1/orders/{id}/apply-billing-rules` — 结账时应用规则

**前端（web-pos结账流程）：**
- 结账页面显示服务费明细行
- 最低消费不足时弹出提示：「本桌消费¥180，最低消费¥200，差额¥20」选项

**验收标准**：配置最低消费200元→桌台消费180元→结账触发提示→服务费按8%自动计算→报表可查服务费收入。
