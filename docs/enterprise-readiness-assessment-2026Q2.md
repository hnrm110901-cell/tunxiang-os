## 二、评估范围与方法

3. **不重复全量计数**：本报告**不**替代 V3 的全库行数/端点统计；**补充**的是「关键差距条文是否仍成立」。

## 三、客户场景 vs 屯象能力映射（简表）

| 需求 | 屯象相对位置 | 说明 |

| 集团 + 多租户隔离 | **强** | RLS + Gateway 租户校验；品牌/区域能力需与菜单、价税、报表端到端验收。 |
| 多品牌 / 多业态 | **中强** | 有组织与增长域能力；**菜单模板 + 渠道价 + 品牌发布** 已与 DB 化实现衔接（见第四节）。 |
| 多渠道收入 | **分化** | 堂食/KDS **强**；外卖聚合 **中等（需 SDK 与统一视图）**；零售/部分创新链路 **偏骨架或分期**。 |
| 品质中餐（包厢/宴请/称重/活鲜） | **中强** | 称重/活鲜、宴席相关代码面较全；宴席支付/对公月结等需按客户合同验收。 |
| 供应链（门店 + 央厨 + 配送） | **较 V2 文档显著改善** | 央厨与配送 **当前以 DB Repository 为主路径**（见下节）。 |
| 财务 / 分润 / 加盟结算 | **中强** | P&L/成本/营收、加盟结算等有落地；**域内分账引擎已写，接口挂载与通道对接需闭环**（见下节）。 |
| AI / 数据 / 安全 | **差异化长板** | Agent、叙事与 BI、等保相关能力在 V3 叙述中已强调；本报告不展开评分细节。 |

### 4.2 配送管理（V2/V3：内存）

| **评估** | **条文过时**。线上应以 **Repository** 为准；内存 dict 不应再作为架构结论依据。 |

### 4.3 菜单模板中心（V2/V3：内存 / 与表两张皮）

| 项目 | 内容 |
|------|------|

| **当前实现** | `menu_template.py` 同样保留 **参考/测试**；`menu_routes.py` 使用 **`MenuTemplateRepository`（v095）**，写入 `menu_templates`、`menu_template_dishes` 等；**渠道侧**另有 `channel_menu_items` 等在 `channel_mapping_routes`、`pricing_routes` 等中的 SQL。 |
| **评估** | **「模板中心纯内存」已不成立**。应区分：**模板主数据（v095）** vs **渠道菜单/定价（多路由）**，分别做集成测试。 |

### 4.4 资金安全 / 利润分账（V2/V3：零代码）

| 项目 | 内容 |
|------|------|
| **历史条文** | V2/V3：**零代码** / 需新建分账引擎。 |
| **当前实现** | 存在 **`services/tx-finance/src/services/split_engine.py`（v100）**：`profit_split_rules`、`profit_split_records` 等 **SQL 持久化** + 租户上下文；**`api/split_routes.py`** 定义 `/api/v1/finance/splits/*` 共 7 个端点。 |
| **接线缺口** | **2026-04-02 起** `services/tx-finance/src/main.py` 已 **`include_router(split_router)`**；若经 **Gateway** 统一出口，仍需确认路由是否转发到 tx-finance。 |
| **评估** | **「零代码」不再成立**；准确表述应为：**「分账域内引擎 + HTTP 已挂载；待支付通道分账 API/回调/对账与业务规则落地」**。 |

### 4.5 供应商门户（V2：纯内存）

| 项目 | 内容 |
|------|------|
| **历史条文** | V2：纯内存。 |
| **当前实现** | `supplier_portal_service.py`：**存在表则走 DB**，表缺失时 **降级内存** 并记录 WARNING。 |

## 五、修订后的「致命差距」清单（以代码为准）

下列表述替代 V2「五大致命」/V3「三大致命」中 **已被代码否定的条目**：

| 序号 | 差距主题 | 状态 | 建议动作 |
|------|----------|------|----------|
| ~~1~~ | 中央厨房全内存 | **已缓解（DB 化）** | 环境验证 v062 + 回归测试 |
| ~~2~~ | 配送全内存 | **已缓解（DB 化）** | 环境验证 v096 + 路由测试 |
| ~~3~~ | 菜单模板纯内存 | **已缓解（Repository）** | 环境验证 v095 + 与渠道发布联调 |
| **A** | 分账 **对外可用性** | **tx-finance 已注册** | 确认 **Gateway** 转发；联调支付机构分账与结算 |
| **B** | 分账 **业务闭环** | **待加强** | 规则引擎与真实流水、差错账、审计报表一致 |
| **C** | 外卖 / 全渠道深度 | **仍属短板带** | 聚合适配器、统一订单中心、配送运营 |
| **D** | 零售 / 部分创新业态 | **骨架或分期** | 按产品线单独立项 |
| **E** | 边缘离线 / 全端 PWA | **中等** | mac-station 本地 PG、多终端 SW 策略 |

**结论**：原「供应链三大件全靠内存」的 **致命级判断需降级**；**真正的当前焦点** 转向：**分账服务接线 + 通道侧合规落地 + 全渠道与零售补齐**。

## 六、SaaS 切换与稳定上线：可行性分级

| 切换策略 | 可行性 | 条件 |
|----------|--------|------|
| **单域试点**（如仅 POS+KDS+会员） | **高** | 数据迁移方案、培训、并行期对账 |
| **多域不含复杂分账** | **中高** | 财务口径确认；可选先用手工/线下分润补位 |
| **集团全渠道 + 加盟分润 + 央厨配送一体切换** | **低（不建议大翻牌）** | 需完成第五节 A/B + 全链路压测与法务合规 |
| **稳定交付定义** | **建议明确 SLA** | 含支付成功率、对账差异阈值、离线策略、回滚预案 |

## 九、一句话结论

> **屯象OS 在供应链中枢（中央厨房、配送）与菜单模板中心上，已较 2026-Q1 文档所描述的「内存致命缺口」大幅收敛；当前更关键的上线风险集中在「分账能力是否对外可用 + 支付通道与业务规则闭环」，以及外卖/零售等分域成熟度。**  
> **替换传统 SaaS 建议采用分阶段切换；文档与营销口径应及时与源码对齐，避免低估已交付能力或高估未接线模块。**
