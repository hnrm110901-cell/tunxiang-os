## 一、天财「餐饮7」产品边界摘要（来自《说明》Sheet）

| 维度 | 正餐行业版 | 正餐企业版 |
|------|------------|------------|
| 用户画像 | 标准化程度高的正餐 | 大型及有特色经营诉求 |
| 定制开发 | 不支持 | 支持 |
| 增值能力 | 行业版**不支持增值功能** | 支持增值模块 |
| 明确排除（行业版说明中写出） | 不支持快餐模式、智慧商街、自助点餐机、**AI** 等 | 企业版承载 |
| 业态侧重 | 正餐、家常菜、火锅、烧烤；**不支持自助餐、粤菜**（说明原文） | 更广 |

**解读**：天财用**版本矩阵**切分能力；屯象用**一套 Web + 配置/租户**扩展。对标时要区分「行业版必有」与「企业版/增值」项，避免把可选增值全部算作行业基线。

## 二、终端与产品线矩阵差距

天财门店清单按**独立产品线**拆分 Sheet；屯象对应多为 **单一代码库多路由**（`web-pos` / `web-kds` / `web-crew` / `web-reception` / `h5-self-order` / 小程序）。

| 天财产品线（门店清单） | 规模（约） | 屯象对应 | 差距说明 |
|------------------------|------------|----------|----------|
| 登录 / 用户桌面 | ~115 行 | `web-pos` 设置、工作台 | **企业微信登录**、**员工卡**等账号体系对接需核对；指纹等依赖硬件 SDK |
| 前台 POS（收银、录单模式） | ~369 行 | `web-pos` | 功能**深度**见第三节 |
| 前台 POS（**快餐模式**，仅企业版） | ~192 行 | `QuickCashierPage` 等 | 天财「快餐模式」为**完整平行交互**；屯象需评估是否覆盖同等结账/取餐流程 |
| **智能出品**（KDS） | ~139 行 | `web-kds` | **展示与标识规则极细**（见第四节） |
| **移动银台** / **移动银台 Pro** | ~168 + ~86 行 | `web-crew`（服务员 PWA） | 天财区分标准/Pro（人均消费、快速下单、简约模式等）；屯象需逐项对照 **Pro 级能力** |
| **Android 平板** / **iOS 平板** | 各 ~190 行 | 浏览器 / WebView | 天财有**自助/服务员模式、绑定客位、演示模式、图片更新**等平板专属配置；屯象多为通用 Web，**专用平板壳与配置项**可补强 |
| **速点**（仅企业版） | ~57 行 | 无独立产品名 | 偏**轻量点餐/档口场景**的独立客户端；屯象若无对等 SKU，标品竞标时算缺口 |
| **智慧商街**（仅企业版） | ~60 行 | 无 | **美食街/多商户档口合一收银**类场景；屯象 ontology 偏单品牌门店，**多商户同一 POS 实例**需架构评估 |
| **打印服务【全部】** | ~42 行 | `print_manager` + `TXBridge` | 天财有**打印管理器 UI**（多实例、导入导出、打印机类型列表、请求视图）；屯象**门店侧可视化打印中心**与**驱动/网口/串口**全矩阵需对齐 |
| **管理器【全部】** | ~21 行（表中多为空占位） | 边缘/门店运维 | 需结合天财实施文档理解具体管理器能力；当前表内有效行少 |

## 三、POS（正餐录单）（天财 ~363 条有效路径级功能 vs 屯象）

以下按天财清单中**出现且屯象侧薄弱或未产品化**的类别归纳（**非**完整枚举 369 条）。

### 3.1 押金、存酒、复杂账务

| 天财能力（归纳） | 屯象现状 | 差距 |
|------------------|----------|------|
| **押金**全流程（主功能下大量子项，清单中占比较多） | `CreditPayPage`、企业挂账、储值等与「押金」语义部分重叠；**餐桌/物品押金**独立台账需核实 | **押金专用模型**（冻结/退还/转消费）与天财对齐度不明，易成中餐/宴会场景缺口 |
| **存酒**（用户桌面 1 行 + 后台营业管理「存酒」模块） | 适配器注释中出现 `wine` 状态；**未见一等公民「存酒管理」产品模块** | **存酒寄存、支取、期初结余、与会员/桌台绑定**通常需独立 UX + API |
| **协议单位 / 挂账**（总部「协议单位」Sheet；蓝图曾规划） | 企业挂账、打印模板含挂账单等**有雏形** | **协议单位主数据、额度、签单人、对账还款**与天财总部 Sheet 21 项流程需逐项闭合 |
| **最低消费补齐、服务费统计**等 | 会员等级有 `min_spend` 等字段；**账单级最低消费/服务费规则引擎**需对照 | 报表与收银规则若未完全打通，会出现「能算不能管」 |

### 3.2 业态与模式

| 天财 | 屯象 | 差距 |
|------|------|------|
| **自助餐**、**粤菜**等在说明中写「行业版不支持」——企业版/业态包 | 通用点餐模型 | 若签粤式/自助客户，需**专属流程**（计时、押金、出品档位） |
| **宴会**相关（清单中有宴会相关路径） | `banquet_*` 服务、通知单打印等 **🟡**（企业级差距报告） | 深度：排菜、场次、定金、分席结账 |
| **外卖**（堂食标识、第三方/自建外卖标识等） | `omni_channel` **🟡**、多适配器 | **平台菜单双向同步、估清同步、骑手对接**与天财价值说明中的「闭环话术」仍有距离 |

### 3.3 操作体验与快捷键

- 天财为 **Windows 客户端思维**：大量 **Ctrl+F1…**、**F1…** 快捷键、**全屏/锁定/应急指南**等。
- 屯象 **Web POS**：需依赖浏览器与壳层；**快捷键体系、离线应急、打印状态面板**若未做到同级，老收银员会感知「不顺手」。

### 3.4 其它高频点

- **实时库存**入口在 POS 主菜单（天财）；屯象有 `InventoryAlertBanner`、`SoldOutPage` 等，需确认是否达到「一键查看可售库存」同级。
- **训练模式 / 演示模式**：平板与移动银台多处出现；屯象 **POS 演示/沙箱**若有缺口，培训与售前会吃亏。

## 四、智能出品（KDS）颗粒度差距

天财「智能出品」Sheet 将能力拆到 **模式 × 页面 × 功能 × 子功能**（约 **139** 行），典型子项包括：

- 大类/小类筛选、汇总/明细切换、**多规格标识**
- **超时预警颜色 / 超时颜色 / 催单变色闪烁**
- **赠单、加单、叫起、套餐名、堂食/第三方外卖/自建外卖/外带**等标识
- **做法、备注、客位、服务员、点菜宝**等显示维度
- **触摸屏操作**（起菜、划菜、等叫、恢复等）细分

**屯象** `web-kds` 已有泳道、暂停、抢单、档口等（差距报告 **A4 ✅**），但对照天财清单时，差距常落在：

1. **标识与着色规则**的可配置项数量（催单/超时/渠道/赠单等多维同时满足）。  
2. **按客位 vs 按菜品**视图切换及厨房习惯细节。  
3. **与 POS 侧「叫起/等叫」**状态机的 100% 同步与回显。

## 五、总部后台（老菜谱 ~403 行 / 门店菜谱 ~164 行）

### 5.1 一级菜单分布（合并单元格向下填充后统计）

| 一级菜单 | 行数约 | 屯象主要映射 | 差距要点 |
|---------|--------|--------------|----------|
| **营业管理** | ~160 | `tx-trade` / `tx-menu` / `tx-growth` / `web-admin` | **打折促销**子树极大（天财 ~74 行级别），对应**营销方案类型、互斥、执行顺序**（参见 `pinzhi-benchmark` 已写品智对标，天财同类） |
| **系统管理** | ~118 | `tx-org` / 配置 / 审计 | **设置**项极多（~96）：MAC 绑定、数据传输、数据整理、查询导入——**实施与运维工具链** |
| **档案管理** | ~86 | `tx-org` / `tx-menu` | **品项档案、营业档案、结算方式、客位档案、设备档案**等与企业级主数据完备度相关 |
| **系统对接** | ~33 | `shared/adapters` | 天财作为成熟 SaaS，**对接清单与认证生态**更完整；屯象适配器存在但**商业认证数量**需持续补 |
| **权限管理** | ~5 | RBAC / RLS | 体量小但**总部 vs 门店 vs 直通车**分权模型要一致 |

### 5.2 门店菜谱 Web（新后台）

- 一级分布：**档案管理、系统设置、营业管理、出品管理、营销活动**等。
- **集团菜品库 / 菜谱方案 / 门店菜谱 / 门店菜品管理 / 菜品分组** — 屯象有品牌发布、菜单工程化等，**「菜谱方案」式批量下发与门店微调**仍需与 `pinzhi-benchmark` 中「三级发布」一起看作持续课题。

## 六、报表中心（天财 ~296 张具名报表）

天财「报表中心」Sheet 中 **有名称的报表约 296 条**，分类上覆盖：

- **营业统计**（营业情况汇总/明细、经营指标走势与对比、时段统计、员工效能产值、门店指标等）
- **财务核账**
- **决策分析**
- **增值报表**

**屯象** `tx-analytics` + `web-admin` 若干看板（含实时、HQ、菜品分析等）与 **Agent/BI** 能力在架构上可超越单表报表，但**对标天财的「报表 SKU 清单」**时：

| 差距类型 | 说明 |
|----------|------|
| **报表数量与命名** | 客户常按「有没有这张表」验收；296 vs 现有前端报表页 **数量级差距** |
| **固定格式导出/打印** | 财税、店长习惯 PDF/Excel 固定模板 |
| **增值报表商业包装** | 天财单独标注增值；屯象需对应 SKU 与权限 |

## 八、《产品功能价值说明》对照（POS / CRM / SCM）

### 8.1 POS 价值点中屯象需加强的「话术级」能力

- **组织机构 + 门店分组 + 档口**：屯象有 ontology，**实施默认包**是否达到天财「一上线就能按区域/管理人分组」需产品化封装。  
- **权限**：天财细分 **门店后台 / 门店前台 / 总部后台 / 总部报表 / 直通车用户**；屯象 RBAC 需 **角色模板** 与天财分类映射。  
- **营业档案**：市别、区域、客位、结算方式、异常原因、订单备注等 **主数据 + 下发** — 对照 `tx-org` / `tx-trade` 是否 **可批量下发、可门店覆盖**。  

### 8.2 CRM / 智慧餐厅

- **排队**：公众号拉新、会员标签、**超时返券**、**预点菜** — `web-reception` + `tx-growth` 需验证是否达到价值说明中的 **运营闭环**。  
- **自营外卖 + 配送**：达达/顺丰等 **配送商对接** 与 **20%～30% 私域外卖** 运营指标 — 属于商务+产品双维度。  
- **美团/饿了么/抖音**：接单、厨房打印、菜单同步、估清同步 — 对齐 `omni_channel`、适配器路线与 **上线门店数量**。  
- **扫码点餐**：多模板（火锅先锅底、快餐、茶饮）、**多人同单可见** — 屯象有 `collab_order`；**模板数量与业态预设**可继续加厚。  

### 8.3 SCM

- **云库存基础版**：一键录入、断网、上下限预警、盘点、会计期、自定义报表组件等 — 对照 `tx-supply`：**移动 EDI、语音盘点、供货商 EDI、门店间移动调拨** 等在价值说明中强调的点，与企业级差距报告中 **供应链移动端、供应商门户** 缺口一致。

## 九、屯象相对天财的**优势或差异化**（简要，便于对内对齐）

- **AI-Native**：Agent、折扣守护、智能排菜、稽核等 — 天财说明中行业版 **明确不写 AI**；此为屯象长期卖点。  
- **统一技术栈**：React + 微服务 + RLS 多租户，避免多套客户端分叉维护。  
- **小票/打印模板编辑器**、**支付 Saga/幂等**、**三单匹配** 等已在 `gap-analysis-enterprise-benchmark-2026Q1-v3` 标为强项。  
- **Mac mini 边缘 + 同步引擎** 路线与天财纯云 SaaS 不同，**离线优先**可打成差异化（需交付上证明稳定性）。
