# 屯象OS 餐饮行业菜单与 UI 架构方案（对标全球 SaaS/Agent 产品）

> 版本：v1.0  
> 日期：2026-04-12  
> 目标：在不推翻现有代码资产的前提下，给出一套“可落地、可演进、可智能化”的菜单与 UI 架构蓝图。

---

## 1. 全球代表性方案扫描（SaaS + Agent）

> 说明：以下仅选取“产品结构与 UI 架构”层面可直接借鉴的样本，而非做市场份额结论。

### 1.1 餐饮 SaaS 代表

1. **Toast（美国）**  
   核心特征：POS + KDS + 多门店菜单管理 + 营销 + 人员管理 + 供应商/成本工具一体化。  
   UI 启示：
   - “门店执行端（POS/KDS）+ 总部管理端（多店分析）”双工作台并行；
   - 经营数据、营销动作、团队流程在同一产品系统内闭环。

2. **Oracle Simphony（全球）**  
   核心特征：云端开放 API、跨触点菜单/促销同步、前厅-后厨-后台贯通。  
   UI 启示：
   - “菜单/价格/促销”必须是跨终端同源配置；
   - 厨房通信与前台操作是同一个交易链路，不应成为割裂系统。

3. **Square for Restaurants（美国）**  
   核心特征：支付、订单、会员回访、团队与资金管理聚合。  
   UI 启示：
   - 通过“能力中心（Capabilities）”组织菜单，比按系统边界更贴近运营心智；
   - 将支付、会员、团队排班放到可联动的流程 UI 中。

4. **Lightspeed Restaurant（全球）**  
   核心特征：以门店效率和多国可扩展为目标，强调 Tableside、KDS、库存、洞察、AI。  
   UI 启示：
   - 快速操作 + 深度洞察分层；
   - “主流程（点单/收银）极简，分析层可深钻”的双层交互。

5. **Olo（美国连锁数字化）**  
   核心特征：Ordering/Dispatch/Rails/Pay/Engage 模块化，重点打通第三方平台订单与 POS/KDS。  
   UI 启示：
   - 多渠道订单要在一个“订单中枢”中可见可控；
   - 菜单与可售状态（availability）必须支持按品牌/门店/时段动态下发。

### 1.2 Agent 平台代表

1. **Salesforce Agentforce**  
   启示：低代码 + 专业代码双模；Agent 配置中必须显式管理 Topic / Action / Guardrail / Data Source。

2. **ServiceNow AI Agents**  
   启示：要把“Agent 编排、监控、治理、跨系统工具连接”做成平台层，而不是散落在单点页面。

3. **Microsoft Copilot Studio**  
   启示：Agent 生命周期（创建→测试→评估→发布→监控）应在产品 IA 中闭环，且支持 MCP/工具接入。

## 2. 屯象OS现状（架构与代码）

### 2.1 已有基础很强，可直接承接“Agentic UI”

- **四栏 HQ Shell 已成型**：`56 | 220 | 1fr | 340`，包含 IconRail、Sidebar、Main、Agent Console。  
- **菜单已数据驱动**：`MENU_CONFIGS` 已抽离，适合升级为“角色 + 场景 + 状态”驱动。  
- **路径到模块映射机制完备**：`deriveModuleFromPath` 已可做统一 IA 中台。  
- **POS 端已具备完整路由骨架**：桌台、收银、结算、叫号、外卖、训练模式都在一套路由内。  
- **Agent OS 服务成熟**：`tx-agent` 已有 orchestrator / skill registry / monitor / ROI / observability 路由。  
- **硬约束机制存在**：毛利、食安、体验在后端可执行，具备“AI 建议不越线”条件。

### 2.2 当前问题（从 IA/交互角度）

1. **菜单项规模过大**：同层级项目过多，决策负担大。  
2. **按模块组织 > 按任务组织**：店长/运营实际高频任务横跨多个域。  
3. **Agent 入口存在但未形成“工作流闭环”**：建议→审批→执行→追踪还未统一一屏。  
4. **多端视觉与信息密度策略不统一**：HQ、POS、移动端缺少统一“任务优先级 UI 语义”。

## 3. 面向餐饮行业特性的目标 IA（信息架构）

### 3.1 一级导航（总部端）建议改为 8 大工作中心

1. **今日经营**（替代分散 Dashboard）
2. **交易与渠道**（堂食/外卖/团购/企业单）
3. **菜单与定价**（菜品、套餐、渠道价、时段可售）
4. **供应与出品**（库存、采购、BOM、KDS 路由）
5. **会员与增长**（分群、旅程、活动、私域）
6. **门店与组织**（门店配置、角色权限、排班绩效）
7. **财务与分析**（P&L、预算、对账、异常）
8. **Agent 中枢**（编排、策略、审批、效果复盘）

> 原有 finance/org/ops/member/growth 的能力不删，只重组挂载。

### 3.2 二级导航采用“任务分组”而非“功能堆叠”

示例（菜单与定价中心）：
- **上新链路**：新品立项 → 试销 → 复盘 → 正式发布
- **定价链路**：基础价 → 渠道价 → 时段价 → 活动价
- **可售链路**：门店可售 → 缺货替代 → 自动下架规则
- **工程链路**：BOM/毛利模拟 → 厨房产能校验

### 3.3 角色化菜单裁剪（必须产品内置）

- **老板**：今日经营、财务与分析、Agent 中枢（审批）优先。
- **区域运营**：交易与渠道、菜单与定价、门店与组织。
- **店长**：门店值守、库存预警、排班与班结、异常处理。
- **收银员/后厨**：仅保留执行流路由，禁入复杂配置页。

## 4. UI 架构方案（HQ + POS + 移动）

### 4.1 统一“3 层页面骨架”

- **L1 导航层**：一级模块 + 二级任务分组 + 搜索/命令面板。
- **L2 决策层**：KPI、异常、建议、审批（Agent 可解释卡片）。
- **L3 执行层**：可直接触发 API 的动作面板（批量/单点）。

### 4.2 HQ 四栏升级建议（保留现有框架）

- 左一 `IconRail`：只保留 8 个一级中心。
- 左二 `Sidebar`：仅显示当前角色可见任务链。
- 中间 `Main`：统一“概览卡 + 待办队列 + 明细表 + 时间线”。
- 右侧 `Agent Console`：升级为“建议、证据、约束、影响、审批”五段式。

### 4.3 POS 执行端 UI 原则

- 仅保留高频 5 流程：开台、点单、加单、结算、异常。
- 所有 AI 建议以“可忽略提示”出现，不抢占收银主流程。
- 与 `TXBridge` 相关动作（打印/钱箱/扫码）统一错误反馈模板。

### 4.4 移动管理端 UI 原则

- 以“待办清单 + 红黄预警 + 一键处理”优先。
- 禁止把 HQ 全量表格搬到手机；只保留可行动信息。

## 5. Agentic UI 关键机制（匹配屯象现有 Agent 架构）

### 5.1 建议卡标准结构（所有 Agent 输出统一）

- `建议内容`（一句话）
- `业务影响`（预计营收/毛利/体验影响）
- `证据链`（数据来源、时间窗、样本量）
- `约束校验`（毛利/食安/体验）
- `动作按钮`（采纳、修改后采纳、驳回、转派）

### 5.2 三态执行模型

- **Auto**：低风险可自动执行（需策略允许）。
- **Confirm**：默认人工确认（推荐）。
- **Block**：触发硬约束时仅可驳回或改方案。

### 5.3 追踪闭环

每条建议都要有：`提议时间`、`处理人`、`处理动作`、`结果指标`、`复盘结论`。  
与 `decision_log`、`agent_roi` 路由打通，实现“可复盘运营”。

## 6. 与现有代码的落地映射（建议实施清单）

### Phase A（2~3 周）：IA 重组不改业务逻辑

1. 重构 `MENU_CONFIGS` 为：
   - `module -> task_groups -> items`
   - 增加 `roles`, `scenes`, `priority`, `badges` 字段
2. `SidebarHQ` 加入：
   - 角色过滤
   - 高频任务置顶
   - 搜索跨组高亮
3. `IconRail` 改为 8 中心并支持“未处理数”徽标。

### Phase B（3~4 周）：Agent Console 产品化

1. 将 `AgentConsole` 升级为建议卡流；
2. 接入 `tx-agent` 的 monitor/plan/roi/observability 路由；
3. 形成建议审批与回写闭环。

### Phase C（4~6 周）：多端统一语义

1. 提炼一套 Token（优先级、风险色、动作级别）；
2. POS 与 HQ 共用“异常卡片/建议卡片”组件语义；
3. 建立跨端交互一致性基线（快捷键、批量操作、反馈样式）。

---

## 7. 关键指标（衡量 UI 架构是否成功）

1. **运营效率**：高频任务完成时长下降（目标 25%+）。
2. **菜单治理效率**：新品发布链路时长下降（目标 30%+）。
3. **异常处理速度**：从告警到处置时长下降（目标 35%+）。
4. **Agent 采纳质量**：建议采纳率与采纳后收益提升。
5. **培训成本**：新店长上手时间下降（目标 40%+）。

---

## 8. 结论

屯象OS当前代码基础已经具备“从功能型后台升级为 Agentic 经营工作台”的条件。建议采取“**先 IA 重组、再 Agent 闭环、后多端统一**”三步走：

- 先把菜单从“系统视角”改为“餐饮任务视角”；
- 再把 Agent 从“能力展示”升级为“可执行流程”；
- 最后在 HQ/POS/移动端统一操作语义，形成可复制的门店数字化操作体系。
