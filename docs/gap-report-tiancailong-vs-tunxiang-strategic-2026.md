## 一、双方定位：必须先对齐的「比较前提」

| 维度 | 天财商龙餐饮 SaaS（以餐饮7 为代表） | 屯象OS |
|------|--------------------------------------|--------|
| 商业形态 | 成熟 **标品 SaaS + 版本矩阵**（行业版/企业版/增值模块），长周期客户基数大 | **AI-Native 经营操作系统**，替换多套旧系统，强调 Ontology + Agent + 边缘智能 |
| 产品表达 | **功能清单极厚**（POS 数百路径、报表近三百张），适配国情餐饮细分业态 | **域服务 + Web 多端一套代码**，能力在微服务与前端路由中展开，清单需「翻译」后才能对标 |
| 终端现实 | 传统 **Win 客户端 + 多专用端**（移动银台、平板、打印管理器等）成熟 | **Web + 安卓 WebView + Mac mini 边缘**；外设走商米与 TXBridge，**不追求 1:1 复刻 Win 快捷键体系** |
| AI | 行业版说明中明确 **不包含 AI**；企业版/增值另议 | **Master + Skill Agent、边缘 Core ML + 云端推理**写入架构宪法，与业务闭环绑定（如折扣守护、稽核） |

**结论**：用「天财 Excel 行数」直接衡量屯象完成度会**系统性偏高估天财、偏低估屯象**——因为天财把每个菜单叶子都列成一行，而屯象很多能力在共享引擎里。反过来，**客户采购与店长验收**仍常按「有没有天财那套」问，所以：**商业上必须覆盖的「表意等价」不能省，实现形态可以不同。**

## 二、连锁餐饮与技术的几条硬趋势（影响「该保留什么」）

下列判断偏**中国大陆连锁正餐与火锅、烧烤、家常菜**主战场，省略空话。

1. **人力与熟练工短缺**：前台要「少培训、少按错、高峰不断档」——等价于 **稳定收银、清晰 KDS、移动点餐、扫码点餐**，而不是更多报表。  
2. **平台抽佣与流量焦虑**：品牌要做 **私域会员、自营外卖、排队/预点**，与美团饿了么抖音 **对接与估清同步**仍是**刚需**，和天财价值说明中的话术一致。  
3. **食安与合规压力**：原料效期、留痕、异常操作可追溯——与屯象 **三条硬约束 + Agent 留痕**方向一致，**传统 SaaS 靠配置与人工**，屯象可用 **自动化校验** 差异化，但**底账与单据**仍要对齐监管与财务习惯。  
4. **多品牌、加盟与分润**：扩张期客户要 **加盟档案、合同费用、分账/对账**；天财有独立模块，屯象需在 **tx-org / tx-finance** 持续产品化，而非只做 API。  
5. **对 AI 的期待回归务实**：市场从「讲故事」转向 **省人、少错、快响应**—— copilot 问数、异常折扣预警、备货建议、排班辅助等 **可度量场景**优先；**不能替代**打印稳定、钱箱准确、断网可卖。  
6. **云 + 边缘**：弱网门店、高峰延迟——屯象 **Mac mini 本地 PG + 同步**若交付稳定，是**长期差异化**；天财路线以云端为中心，边缘能力不在同一叙事。

## 三、天财侧：哪些能力与模块「市场上仍要保留」（表意必须覆盖）

以下指**客户与实施仍会拿来对标**的能力域——**不是**要求屯象做一模一样的 Win 客户端，而是**业务结果等价**。

### 3.1 门店交易与厨房（现金牛）

- **正餐 POS 主路径**：开台、点菜、加单、改价/折扣授权、结算、多种支付、发票与钱箱、反结/冲红类异常（权限化）。  
- **押金、最低消费、服务费、宴会/排菜、团购验券**：中餐大单店高频；天财在清单中占比大，屯象需在 **规则引擎 + 订单模型** 上补课（见功能差距明细）。  
- **存酒**：独立台账 + 会员/桌台关联；当前屯象**明显薄**，属「天财有、客户会问」的典型。  
- **KDS**：不仅是「能出单」，而是 **超时/催单/渠道/赠单/叫起** 等标识与配色——厨房肌肉记忆。  
- **打印**：网口/串口/驱动、分单/总单、模板——中国餐饮**强依赖**；屯象已有后端与 TXBridge 方向，**门店侧打印中心 UI 与运维体验**要对标天财「打印服务」的**可实施性**。  
- **移动点餐 / 平板**：天财拆多条产品线；屯象用 **web-crew + 平板 WebView** 可接受，但 **Pro 级能力**（快速下单、人均、演示模式等）要收敛进路线图。

### 3.2 总部与经营分析（续约与老板满意度）

- **主数据**：组织、门店、部门、员工、权限（总部/门店/报表/移动端分权）、营业市别、客位、结算方式——**扩张期硬需求**。  
- **菜谱体系**：集团菜库、方案下发、门店菜谱与价格——天财「老菜谱/门店菜谱」双轨；屯象 **品牌发布与模板**要对齐**实施话术**。  
- **营销与促销**：方案类型多、互斥与执行顺序——连锁**真实复杂度**在 tx-growth，天财「营业管理-打折促销」条目多即是证据。  
- **报表**：近 **300 张具名报表**是天财壁垒之一；客户会问「有没有这张表」。屯象不必 **1:1 抄表**，但需 **两类答案**：  
  - **财税/店长刚需的固定表**（少量、可打印）；  
  - **其余用指标库 + 自助探索 + NLQ/Agent** 替代，并在商务上**明确写进方案**，避免验收扯皮。  

### 3.3 供应链与加盟（利润与规模）

- **进销存、成本卡、盘点、调拨、采购与供应商**——与天财 SCM 价值说明一致；移动端订货/收货、供货商协同属**加分刚需**。  
- **加盟、协议单位、挂账、计件提成**——规模化连锁必问；屯象已有碎片能力，要**闭环到 UI + 报表 + 对账**。

### 3.4 可弱化「形式」、不可弱化「结果」

- **Windows 快捷键密度**：可弱化；**高峰操作步数与误触率**不可弱化——用 Web 与壳层优化补。  
- **独立 exe 数量**（速点、商街等）：多数客户不需要全部；但**对应业态**（美食街多商户、极简档口）若主攻，需**单列 SKU 或配置包**。  
- **报表张数**：形式上可弱化；**监管与老板要的少数关键表**不可弱化。

## 四、屯象侧：结合 AI 智能体与架构，应优先开发的路径

与 `CLAUDE.md` 一致：**交易可靠优先，Agent 不替代打印与钱箱**。

### 4.1 第一梯队（不做会丢单、难验收）

1. **中餐账务专模**：存酒、餐桌押金、协议单位与挂账额度、最低消费/服务费与报表勾稽。  
2. **外卖/平台**：接单、厨房分流、**菜单与估清同步**——对齐天财价值说明中的「闭环话术」。  
3. **KDS 可配置规则**：颜色/闪烁/渠道/叫起与 POS 状态机严格一致。  
4. **打印与门店运维**：打印管理可视化、任务队列、重打与对账辅助。  
5. **「报表验收」策略落地**：选定 **P0 固定报表清单**（如 20～40 张）+ 指标字典 + Admin/Agent 问数，写进**交付标准**。

### 4.2 第二梯队（与混合架构强相关）

1. **边缘 + 同步**：断网收银、本地 Agent（出餐预测、折扣风险轻量模型）、冲突策略——**差异化于纯云 SaaS**。  
2. **支付与分账**：Saga/幂等已有；**支付通道回调与多方分账**持续对接（企业级差距 V3 已点出）。  
3. **供应链移动与 EDI**：门店调拨/盘点/移动订货、供应商门户——与天财 SCM 移动场景对齐。

### 4.3 AI 智能体：建议「绑定业务 KPI」而非「功能列表+1」

| 方向 | 与天财关系 | 屯象做法 |
|------|------------|----------|
| 折扣/赠菜合规 | 天财靠权限与人工 | **折扣守护 Agent + 留痕**，可审计、可解释 |
| 店长问数 | 天财靠点开多张报表 | **NLQ + 指标语义层**（替代部分报表张数） |
| 备货与损耗 | 天财 SCM + 经验 | **库存预警 Agent + 边缘预测**（与 sync 数据质量绑定） |
| 客服与差评 | 传统 CRM 模块 | **智能客服 Skill** 与会员旅程打通 |
| 排班与人效 | 计件提成 3.0 等 | **与 tx-org 考勤/绩效数据**联动，避免孤立聊天机器人 |

**实事求是**：Agent 的价值在 **降低异常与决策延迟**；若 **主数据与订单事实层**不稳，Agent 只会放大噪声。故 **P0 仍是交易与主数据**，AI 是 **放大器**。

## 五、差距总表：保留 / 替代 / 补强（战略视角）

| 类别 | 天财模块/能力（表意） | 屯象策略 |
|------|------------------------|----------|
| **必选对标** | POS 主链、KDS、支付结算、会员储值券、基础供应链、组织权限、核心报表 | **功能等价 + 验收清单**；细节见 V2 对照表 |
| **形态替代** | 296 报表、Win 快捷键、多 exe | **固定表 + BI/Agent**；Web 快捷键与壳层；**一套 Web 多路由** |
| **明显缺口（须开发）** | 存酒、押金专模、协议单位闭环、部分业态包（快餐平行流程、智慧商街类） | **按目标客户业态排期**；不必一次全做 |
| **屯象长板（不必矮化）** | 行业版不含的 AI、边缘智能、Saga/幂等、小票可视化编辑、三单匹配等 | **商务与架构叙事**明确写清，用于抬标与溢价 |
| **共同长期战** | 平台对接深度、财税合规、多租户实施效率 | 行业共性；**适配器 + 开放 API**与实施工具（如门店克隆）持续投入 |

## 六、结论（给决策用的一页纸）

1. **天财商龙 SaaS 里「该保留」的**，本质是 **二十年餐饮实施沉淀出的验收语言**：厚 POS、厚 KDS、厚报表、厚营销规则、厚供应链主数据——**客户心智仍在，短期不会消失**。  
2. **屯象不必复制天财的形态**，但必须交付 **同一类经营结果**；缺口最大的往往不是「有没有点菜」，而是 **存酒/押金/协议挂账/报表 SKU/平台同步** 等**细颗粒闭环**。  
3. **AI 智能体**应聚焦 **合规、问数、预警、少人化**，与 **Mac mini 边缘** 结合形成天财**行业版架构难以短期模仿**的壁垒——但前提是 **P0 交易与数据底座**过硬。  
4. **技术上**：用 **一份差距明细 + 一份 P0 报表清单 + 分业态里程碑** 管理预期，比「逐行抄天财 Excel」更符合屯象 **操作系统** 定位与投入产出。
