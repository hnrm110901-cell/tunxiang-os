### Sprint 1 — 店长效率提升（第 2-3 周）

> 目标：**智能备料建议 + 全渠道沽清**，解决店长和厨师长最痛的两个手工环节

#### 1A. 智能备料建议单（1.5 周）

**现状**：BOM 版本管理完整，但无自动建议
**要做**：基于（明日预订量 + 历史同期销量 + BOM 理论用量）自动生成备料单

| 文件 | 变更 | 说明 |
|------|------|------|
| `services/prep_suggestion_service.py` | 新建 | 核心算法：预测销量 × BOM用量 × 损耗系数 = 建议采购量 |
| `api/prep_suggestion.py` | 新建 | 3 个端点：生成建议 / 确认 / 历史查询 |
| `pages/sm/PrepSuggestion.tsx` | 新建 | 店长移动端：建议单列表 + 一键确认 + 微调数量 |
| `main.py` | 修改 | 注册路由 |

**算法逻辑**：
```
建议采购量 = (明日预订桌数 × 平均人数 × 菜品点击率 × BOM单位用量)
           + (历史同日销量均值 × BOM单位用量 × 安全系数1.1)
           - 当前库存量
           + 损耗补偿量(损耗率 × 总需求)
```

**验收标准**：
- 店长早上 8:00 收到备料建议推送
- 一键确认后自动生成采购申请单
- 历史回看：建议量 vs 实际消耗偏差 < 15%

#### 1B. 一键全渠道沽清（0.5 周）

**现状**：库存预警存在，但沽清不联动
**要做**：厨师长一键触发 → POS + 美团 + 小程序同步下架

| 文件 | 变更 | 说明 |
|------|------|------|
| `services/soldout_service.py` | 新建 | 沽清调度器：写入本地状态 + 异步通知各渠道 |
| `api/soldout.py` | 新建 | 3 个端点：触发沽清 / 恢复上架 / 当前沽清列表 |
| `pages/chef/Soldout.tsx` | 新建 | 厨师长看板：菜品卡片 + 一键沽清按钮 + 状态同步指示 |
| `services/pos_service.py` | 修改 | 增加 `set_dish_soldout()` 接口调用 |

**验收标准**：
- 厨师长点"沽清"后，POS 菜单 30 秒内标灰
- 美团/小程序（如已对接）同步下架
- 恢复上架同样一键生效

### Sprint 2 — 楼面 + 厨房升级（第 4-5 周）

> 目标：**桌态时长预警 + 出品质量看板**，让楼面和厨房从"靠盯"变成"靠系统"

#### 2A. 桌态时长预警 + 翻台预测（1 周）

**现状**：排队叫号完整，无时长监控
**要做**：每桌入座后开始计时，超阈值预警，预测翻台时间

| 文件 | 变更 | 说明 |
|------|------|------|
| `services/table_monitor_service.py` | 新建 | 桌态引擎：入座计时 + 阈值预警 + 翻台预测(基于历史均值) |
| `api/table_monitor.py` | 新建 | 4 个端点：桌态总览 / 预警列表 / 翻台预测 / 等位预估 |
| `pages/floor/TableMonitor.tsx` | 新建 | 楼面看板：桌态矩阵(颜色=时长) + 预警弹窗 + 等位预估 |

**预警规则**：
- 🟢 正常：用餐 < 该餐段均值
- 🟡 关注：用餐超均值 20%（如正餐超 90 分钟）
- 🔴 需干预：用餐超均值 50%（如正餐超 110 分钟）

**验收标准**：
- 楼面经理大屏实时看到每桌用餐时长和颜色
- 超时桌自动弹出"建议主动关怀"提醒
- 等位顾客能看到预估等待时间

#### 2B. 厨师长出品质量看板（1 周）

**现状**：菜品数据分散在多个模块，厨师长无专属视图
**要做**：聚合退菜 + 出品时长 + 顾客评价 → 出品质量评分

| 文件 | 变更 | 说明 |
|------|------|------|
| `services/dish_quality_service.py` | 新建 | 聚合引擎：退菜率 + 平均出品时长 + 差评关联 + 质量评分 |
| `api/dish_quality.py` | 新建 | 4 个端点：质量总览 / 菜品排名 / 趋势分析 / 预警列表 |
| `pages/chef/DishQuality.tsx` | 新建 | 厨师长看板：质量评分卡 + 退菜Top5 + 出品时长分布 + 趋势图 |

**质量评分公式**：
```
菜品质量分 = (1 - 退菜率) × 40
           + (1 - 超时出品率) × 30
           + 好评率 × 30
```

**验收标准**：
- 厨师长登录后首页看到今日出品质量评分
- 退菜 Top5 菜品实时排名（含退菜原因分布）
- 出品时长超标菜品自动标红

### Sprint 3 — 服务闭环 + 老板穿透力（第 6-7 周）

> 目标：**差评处理闭环 + 老板穿透力看板**，补上诊断报告里的管理闭环

#### 3A. 差评处理闭环（1 周）

**现状**：情感分析存在，无 SLA 管理和闭环追踪
**要做**：差评 → 自动派单 → 处理计时 → 回访验证 → 关闭归档

| 文件 | 变更 | 说明 |
|------|------|------|
| `services/complaint_service.py` | 新建 | 投诉工单引擎：创建 → 派单 → 处理 → 回访 → 关闭 5 态 |
| `api/complaints.py` | 新建 | 6 个端点：工单 CRUD + 指派 + 关闭 + SLA 统计 |
| `pages/ComplaintPage.tsx` | 新建 | 投诉看板：工单列表 + SLA 倒计时 + 处理流程时间轴 |

**SLA 规则**：
- 差评工单 30 分钟内必须指派责任人
- 2 小时内必须首次回复/处理
- 24 小时内必须关闭或升级
- 超时自动企微提醒店长 + 区域经理

#### 3B. 老板穿透力看板（1 周）

**现状**：数据展示平铺，缺乏"谁在蒙混过关"的穿透能力
**要做**：同类门店标杆对比 + 异常排名 + 系统 ROI 自证

| 文件 | 变更 | 说明 |
|------|------|------|
| `services/boss_insight_service.py` | 新建 | 老板洞察引擎：门店排名 + 异常检测 + ROI 计算 |
| `api/boss_insights.py` | 新建 | 4 个端点：门店排名 / 异常门店 / ROI 报告 / 趋势对比 |
| `pages/hq/BossInsights.tsx` | 新建 | 老板看板：门店赛马榜 + 异常预警 + 屯象OS价值自证卡 |

**ROI 自证逻辑**：
```
本月屯象OS价值 = 拦截损耗异常次数 × 平均每次挽回¥
               + 备料建议节省的食材¥
               + 沽清避免的退菜损失¥
               + 会员召回带来的增量营收¥
```

**验收标准**：
- 老板看板首屏：全集团今日健康度(分数) + 最佳门店 + 最差门店
- 点击门店下钻：该店 vs 同类标杆的雷达图对比
- "屯象OS 本月为您节省 ¥XX" 价值卡每月自动更新

### Sprint 4 — 会员深度运营（第 8-9 周）

> 目标：**全渠道会员 ID 合并**，这是私域运营的底层基础

#### 4A. 全渠道会员身份合并（2 周）

**现状**：完全不存在，堂食 / 外卖 / 小程序会员各自独立
**要做**：基于手机号 + 模糊匹配建立统一会员画像

| 文件 | 变更 | 说明 |
|------|------|------|
| `models/unified_member.py` | 新建 | 统一会员模型：golden_id + 来源渠道列表 + 合并置信度 |
| `services/member_identity_service.py` | 新建 | 合并引擎：手机号精确匹配 + 姓名模糊匹配 + 消费模式相似度 |
| `api/member_identity.py` | 新建 | 5 个端点：合并建议 / 确认合并 / 拆分 / 统一画像 / 匹配统计 |
| `pages/MemberIdentityPage.tsx` | 新建 | 会员合并管理：待合并列表 + 置信度排序 + 一键合并/忽略 |

**合并策略**：
```
1. 手机号完全相同 → 自动合并（置信度 99%）
2. 姓名相同 + 消费门店重叠 → 建议合并（置信度 80%）
3. 消费模式相似 + 时间段互斥 → 人工确认（置信度 60%）
```

**验收标准**：
- 系统自动识别重复会员并生成合并建议
- 合并后统一画像包含全渠道消费记录
- Customer360 页面展示合并后的完整会员旅程

## 五、风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| G10/正品 API 文档缺失或不稳定 | 高 | S0 延期 | 提前获取 API 文档；准备 mock 数据先跑通流程 |
| 徐记内部审批流程长 | 中 | 全程 | S0 并行推进：技术侧先用 mock，商务侧推审批 |

| 会员手机号数据质量差 | 中 | S4 合并率低 | 先做自动合并（手机号匹配），模糊匹配二期再做 |

## 六、关键结论

1. **诊断报告 12 项需求，其中 6 项已在代码中完整实现**（对账、告警、晨报、会员触达、损耗归因、活动ROI），第一周纯配置即可交付
2. **真正需要新写代码的是 6 项**：备料建议、全渠道沽清、桌态预警、出品质量、差评闭环、会员合并
3. **10 周可全部落地**，第 1 周老板就能看到效果（晨报+对账+告警）
4. 最大的不确定性在**数据源对接**（G10/正品API），建议第一周优先攻克
