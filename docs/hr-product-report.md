## 一、产品定位

屯象OS 人力管理模块（`tx-org` 人力子域 + `tx-agent` 人力 Agent）是连锁餐饮行业**首个将 POS 营业数据、菜品数据、客流数据与人力决策深度融合**的智能人力管理系统。

**核心理念：人力决策不是孤立的 HR 系统，而是建立在 Ontology 全链路数据之上的智能决策。**

## 三、核心产品模块

### 3.1 智能排班引擎

**服务文件**: `services/tx-org/src/services/smart_schedule.py`（1,200+ 行）

这是屯象OS 最核心的人力管理能力——**客流预测驱动的 AI 自动排班**，也是与传统 HR 系统最大的差异化所在。

#### 核心流程（五步法）

```
客流预测 → 人力需求计算 → 贪心+约束满足排班 → 劳动法合规校验 → 技能匹配
```

#### 3.1.1 客流预测模型

`predict_traffic()` 基于多维因子组合预测每小时客流量：

| 因子类别 | 参数 | 示例值 |
|---------|------|--------|
| 星期系数 | 周一~周日 | 0.85（周一）~ 1.30（周六） |
| 天气系数 | 晴/阴/雨/暴雨/雪/酷暑 | 0.55（雪天）~ 1.0（晴天） |
| 节假日系数 | 普通/法定/节前/春节/国庆/情人节 | 0.40（春节长沙）~ 1.60（国庆） |
| 活动系数 | 附近活动影响 | 演唱会 1.15 |
| 时段分布 | 长沙湘菜馆典型时段 | 午高峰 11-13 点占 40%，晚高峰 17-20 点占 39% |

**计算公式**: `日预测客流 = 基础日客流 × 星期系数 × 天气系数 × 节假日系数 × 活动系数`

支持两种粒度：小时级（hour）和班次级（shift）。

#### 3.1.2 人力需求计算

`calculate_staffing_need()` 将客流预测转化为各岗位各小时人力需求：

| 岗位 | 配置标准 | 算法 |
|------|---------|------|
| 服务员(waiter) | 每人负责 4 桌 | `ceil(在场桌数 / 4)` |
| 炒菜厨师(chef_hot) | 每人每小时 25 位客人 | `ceil(客流 / 25)` |
| 冷菜厨师(chef_cold) | 每人每小时 40 位客人 | `ceil(客流×30% / 40)` |
| 点心厨师(chef_dim_sum) | 每人每小时 30 位客人 | 客流>15 时启用 |
| 收银员(cashier) | 每班次至少 1 人 | 固定 1 人 |
| 迎宾(host) | 高峰期至少 1 人 | 高峰期 1 人 |
| 值班经理(manager) | 每班次至少 1 人 | 固定 1 人 |

额外考虑：开餐前 2 小时厨房准备 + 打烊后 1 小时清场。

#### 3.1.3 自动排班算法

`generate_schedule()` 使用**贪心 + 约束满足**算法，在 7 天维度内自动生成排班表：

**劳动法约束（硬约束）**:
| 约束项 | 限制值 | 依据 |
|--------|--------|------|
| 标准工时制周工时上限 | 40 小时 | 劳动法 |
| 综合工时制周工时上限 | 44 小时 | 劳动法 |
| 班次间最少休息 | 11 小时 | 劳动法 |
| 最多连续工作天数 | 6 天 | 劳动法 |
| 月加班上限 | 36 小时 | 劳动法 |
| 未成年日工时上限 | 8 小时 | 劳动法 |
| 未成年夜班截止 | 22:00 | 劳动法 |

**班次定义**（4 类标准班次）:
| 班次 | 时段 | 时长 | 覆盖高峰 |
|------|------|------|---------|
| 早班(morning) | 08:00-15:00 | 7h | 午餐 |
| 中班(middle) | 11:00-19:00 | 8h | 午餐+晚餐前段 |
| 晚班(evening) | 15:00-22:00 | 7h | 晚餐 |
| 全天班(full) | 09:00-21:00 | 10h(含休息) | 午餐+晚餐 |

**排班优化目标**:
1. 满足每小时各岗位最低人力需求
2. 员工技能与岗位需求匹配
3. 尊重员工班次偏好
4. 均衡工时分配
5. 最小化人力成本

### 3.2 考勤引擎

**服务文件**: `services/tx-org/src/services/attendance_engine.py`（736 行）

完整的考勤管理引擎，与排班引擎联动，覆盖打卡、请假、加班、异常检测全流程。

#### 3.2.1 打卡管理

| 功能 | 方法 | 说明 |
|------|------|------|
| 上班打卡 | `clock_in()` | 4 种打卡方式：设备/人脸/APP/手工 |
| 下班打卡 | `clock_out()` | 自动计算工时并配对上班记录 |
| 每日考勤 | `get_daily_attendance()` | 门店全员当日考勤总览 |
| 月度汇总 | `get_monthly_summary()` | 出勤天数/迟到/早退/旷工/加班统计 |

**排班联动判定逻辑**:
- 打卡时自动匹配当日排班班次
- 迟到宽限：5 分钟
- 超过宽限：标记为 `late`（迟到）
- 提前半小时以上到达：标记为 `early`（提前）
- 无排班但打卡：标记为 `unscheduled`（非排班日上班）
- 下班打卡提前超过宽限：标记为 `early_leave`（早退）
- 下班打卡晚超过 30 分钟：标记为 `overtime`（加班）

**事件发布**: 迟到时自动发布 `ATTENDANCE_LATE` 事件，触发 Agent 预警链路。

#### 3.2.2 请假管理

支持 7 类假期，含湖南省地方标准：

| 假期类型 | 默认天数/年 | 带薪 | 说明 |
|---------|------------|------|------|
| 年假(annual) | 5 天 | 是 | 法定年休假 |
| 病假(sick) | 15 天 | 是(打折) | 医疗期 |
| 事假(personal) | 10 天 | 否 | 私事假 |
| 产假(maternity) | 158 天 | 是 | 湖南省标准 |
| 陪产假(paternity) | 20 天 | 是 | 湖南省标准 |
| 婚假(marriage) | 3 天 | 是 | 法定 |
| 丧假(bereavement) | 3 天 | 是 | 法定 |

完整流程：`申请请假 → 余额校验 → 审批 → 扣减余额 → 考勤联动（当日标记为请假）`

#### 3.2.3 加班管理

| 加班类型 | 倍率 | 说明 |
|---------|------|------|
| 工作日加班(weekday) | 1.5x | 延长工时 |
| 周末加班(weekend) | 2.0x | 休息日加班 |
| 法定节假日加班(holiday) | 3.0x | 国定假日加班 |

加班时长上限：0-12 小时/次，月度累计不超过 36 小时。

#### 3.2.4 考勤异常检测

`get_attendance_anomalies()` 自动检测并分级预警：

| 异常类型 | 严重级别 | 描述 |
|---------|---------|------|
| 旷工(absent) | HIGH | 已排班但未打卡 |
| 漏打卡(missing_clock_out) | MEDIUM | 已上班但未打下班卡 |
| 非排班日上班(unscheduled_work) | MEDIUM | 未排班但有打卡 |
| 迟到(late) | LOW | 超过宽限期到达 |
| 早退(early_leave) | LOW | 提前离开 |

高严重级别异常自动发布 `ATTENDANCE_EXCEPTION` 事件，触发 Agent 跟进。

### 3.3 薪资计算引擎

屯象OS 拥有**三层薪资计算架构**，实现从纯函数计算到全链路 DB 批量处理的完整闭环：

| 层级 | 文件 | 职责 |
|------|------|------|
| 纯函数计算层 | `payroll_engine.py` | 基本工资/加班费/提成/全勤奖/工龄补贴/扣款(迟到/早退/旷工) — 无 DB 依赖 |
| 单员工编排层 | `payroll_engine_v2.py` | 组合调用纯函数层，编排单员工月度计算 |
| DB 批量引擎 | `payroll_engine_db.py`（918 行） | 异步 DB 读写 + 月度全店批量计算 + 状态机管理 |

#### 薪资计算全流程

```
1. 读取门店员工列表 + 薪资配置
2. 读取当月考勤汇总(attendance_records)
3. 按薪资方案计算各员工:
   ├── 基本工资 = 月薪 × (实际出勤天数 / 标准工作日数)
   ├── 加班费 = 时薪 × 加班时长 × 倍率(1.5x/2.0x/3.0x)
   ├── 绩效奖金 = 基于 performance_score
   ├── 提成 = 基础 + 推菜 + 开瓶 + 加单
   ├── 全勤奖 = 无迟到/旷工/早退时发放
   ├── 工龄补贴 = 按司龄月数梯度
   ├── 扣款:
   │   ├── 迟到扣款(次数 × 金额)
   │   ├── 早退扣款(次数 × 金额)
   │   └── 旷工扣款(天数 × 日薪 × 倍率)
   ├── 五险一金(个人 + 企业)
   └── 个人所得税(累计预扣法)
4. 生成 payroll_records_v2 草稿
5. 发布 PAYROLL_GENERATED 事件
```

#### 薪资状态机

```
draft → confirmed → paid
```
- `draft`: 计算完成，待确认
- `confirmed`: 已审核确认，待发放
- `paid`: 已标记发放

#### 工资条数据结构（PayrollRecord）

| 字段 | 类型 | 说明 |
|------|------|------|
| base_salary_fen | int | 基本工资(分) |
| commission_fen | int | 提成(分) |
| overtime_pay_fen | int | 加班费(分) |
| bonus_fen | int | 绩效奖金(分) |
| deductions_fen | int | 扣款合计(分) |
| social_insurance_fen | int | 个人五险(分) |
| housing_fund_fen | int | 个人公积金(分) |
| gross_salary_fen | int | 应发工资(分) |
| net_salary_fen | int | 实发工资(分) |

### 3.6 员工深度业务 — 绩效/提成/培训

#### 3.6.1 业绩归因

`calculate_performance_attribution()` — 通过 `waiter_id` 关联 `orders` 表，自动归因员工服务效能：

| 指标 | 数据来源 | 说明 |

### 3.7 门店借调与成本分摊

**服务文件**: `services/tx-org/src/services/store_transfer_service.py`（459 行）

连锁餐饮中高频出现的场景 — 员工跨门店借调时的工时和成本如何准确分摊。

#### 核心能力

| 功能 | 方法 | 说明 |
|------|------|------|
| 创建借调单 | `create_transfer_order()` | 员工/原门店/目标门店/日期/原因 |
| 审批借调单 | `approve_transfer_order()` | 状态流转 pending → approved |
| 工时拆分 | `compute_time_split()` | 按借调日期将考勤工时自动归入对应门店 |
| 成本分摊 | `compute_cost_split()` | **最大余额法**按工时比例分摊薪资(分级精度零误差) |
| 明细分摊表 | `generate_detail_report()` | 单员工各门店工时与成本明细 |
| 薪资汇总表 | `generate_summary_report()` | 按门店汇总所有借调员工成本 |
| 成本分析表 | `generate_cost_analysis_report()` | 实际 vs 预算 + 环比变化 |

**关键算法 — 最大余额法**: 将整数金额(分)按比例分摊到多门店时，采用最大余额法保证分摊后总额与原始总额完全一致，不存在分/厘级误差。

### 3.8 人效指标体系

**服务文件**: `services/tx-org/src/services/labor_efficiency_service.py`（546 行）

对标商龙i人事，建立餐饮行业标准化人效评估体系。

#### 五大核心指标

| 指标 | 计算公式 | 行业基准(目标) | 预警阈值 |
|------|---------|--------------|---------|
| 人力成本占比 | 人力总成本 / 总营收 | 25% | >30% |
| 人均贡献产值 | 总营收 / 人数 | ≥35,000元/月 | <25,000元/月 |
| 人时营业额 | 总营收 / 总工时 | ≥150元/时 | <100元/时 |
| 人时待客数 | 接待顾客数 / 总工时 | ≥1.5人/时 | <1.0人/时 |
| 工作有效性 | 有效工时 / 总工时 | ≥80% | <70% |

#### 综合评分体系

加权计算（人力成本占比 25% / 人均产值 20% / 人时营收 20% / 人时待客 15% / 有效性 20%），四档评级：

| 评级 | 分数区间 | 含义 |
|------|---------|------|
| excellent | ≥90 | 优秀 |
| good | 70-89 | 良好 |
| warning | 50-69 | 需关注 |
| critical | <50 | 严重不达标 |

#### 四角色看板

| 角色 | 看板内容 | 方法 |
|------|---------|------|
| **老板** | 品牌整体人效 + 门店排名 + 成本趋势 | `get_boss_view()` |
| **HR** | 人员编制 + 离职率 + 招聘需求 + 薪酬分析 | `get_hr_view()` |
| **店长** | 本店人效 + 排班优化建议 + 员工绩效 | `get_manager_view()` |
| **员工** | 我的绩效 + 我的考勤 + 我的工资 | `get_staff_view()` |

### 3.9 离职结算

**服务文件**: `services/tx-org/src/services/separation_settlement.py`（484 行）

严格依据《中华人民共和国劳动合同法》第四十六、四十七条，完整实现 N/N+1/2N 经济补偿金计算：

| 离职类型 | 补偿标准 |
|---------|---------|
| 主动辞职(resign) | 0 |
| 过失性辞退(dismiss_fault) | 0 |
| 协商解除(mutual) | N × 基数 |
| 无过失性辞退(dismiss_no_fault) | (N+1) × 基数（含代通知金） |
| 经济性裁员(layoff) | N × 基数 |
| 合同到期不续(contract_expire) | N × 基数 |
| 退休(retire) | 0 |

- **N** = 工作年限（满6个月进1，不满6个月按0.5）
- **基数** = 离职前12个月平均工资
- **上限** = 当地社平工资3倍，且最多12年
- **离职补偿金个税**: 免税额 = 社平工资×3（长沙约295,704元），超出部分按折算月收入单独计税
- 内置长沙 2025 年社平工资数据：98,568 元/年

## 九、技术亮点

1. **金额单位统一为"分"(fen)** — 全系统使用整数运算，避免浮点精度问题，与 PostgreSQL bigint 直接映射
2. **最大余额法成本分摊** — 多门店借调场景下，整数分摊零误差
3. **累计预扣法个税计算** — 严格按税务局要求的年度累计方式计算，避免月末税款波动
4. **多城市社保费率配置** — 支持长沙/北京/上海等城市差异化费率，环境变量可覆盖
5. **事件驱动 Agent 联动** — 考勤异常自动触发 Agent 预警链路，实现"数据 → 检测 → 预警 → 跟进"闭环
6. **客流预测驱动排班** — 业内唯一将 POS 客流数据直接驱动排班算法的系统
7. **RLS 多租户隔离** — 所有 DB 操作强制 `tenant_id` + `SET app.tenant_id`，双重保障数据安全
8. **敏感字段 AES-256-GCM 加密** — 身份证号、银行账号字段加密存储
