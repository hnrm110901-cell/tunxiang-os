### Stage 1：预订入座 `[T+0:00 ~ T+5:00]`

**系统**：web-reception（前台接待系统）

`[操作]` 打开浏览器，访问 `http://reception.store.local`，使用 `demo_admin` 登录。

`[说词]`
> "我们先从最常见的场景开始——顾客提前预订。这是我们的前台接待系统，支持电话预订、小程序预订和现场预订三种来源，都汇总到同一个看板里。"

`[操作]` 点击"新建预订"，填写：
- 顾客姓名：张三
- 人数：4人
- 预约时间：今天晚上 18:30
- 备注：窗边位置

`[展示]` 预订成功后，桌台拓扑图上对应台位变为"预订中"状态（橙色标注）。

`[操作]` 模拟顾客到店：点击该预订记录 → "确认到店" → 选择台位（如 A03） → "分配入座"。

`[说词]`
> "顾客一到，前台一键确认，台位状态从预订变为使用中，后厨和服务员端实时同步。不需要手动传达，不会漏单。"

`[展示]` 桌台拓扑图台位颜色变为"使用中"（绿色），入座时间自动记录。

1. 手机打开 H5 等位页面
2. 填写姓名 + 人数 → 点击"立即取号"
3. 页面显示：当前排队 3 桌，预计等待 15 分钟，号码 #007

`[展示]` 前台看板实时更新等位队列，#007 出现在列表末尾。

`[操作]` 点击叫号按钮（叫 #005），演示微信推送通知。

`[说词]`
> "顾客收到通知后扫码进店，前台一键确认。整个等位过程零人工干预，顾客体验好，前台效率也高。"

### Stage 3：智能点餐 `[T+10:00 ~ T+20:00]`

**系统**：web-pos（POS收银 + Agent预警）

`[操作]` 切换到安卓 POS 主机，打开 POS 页面，选择台位 A03（已入座）。

`[说词]`
> "接下来是点餐环节，也是我们最核心的功能之一。我们的点餐支持搜索、扫码、分类浏览三种方式，但最酷的是这个——"

`[操作]` 点击搜索框，输入："来份招牌菜"（NLQ 自然语言搜索）。

`[展示]` 系统解析语义，推荐本店销量前3的招牌菜，标注毛利率和今日剩余数量。

`[说词]`
> "顾客说'来份招牌菜'，系统自动推荐，不需要顾客背菜单。服务员按一下就加入购物车。"

`[操作]` 选择2道菜加入购物车，然后尝试手动输入一个**超大折扣**（如全单打5折）。

`[展示]` 折扣守护 Agent 触发预警弹窗：
```
⚠️  折扣守护 Agent 预警
折扣率 50% 超过毛利底线（当前毛利将低于 15%）
建议最高折扣：85折
需要店长授权码方可继续
```

`[说词]`
> "这就是我们的折扣守护 Agent。任何可能打穿毛利底线的折扣，系统会自动拦截，要求店长授权。这一条规则每年能帮门店挽回几万元的折扣损失。"

`[操作]` 取消超大折扣，改为合规折扣（9.5折），继续点餐。

`[操作]` 演示菜品推荐：加入一道菜后，右侧出现"搭配推荐"（基于历史订单关联规则）。

`[说词]`
> "搭配推荐是基于真实订单数据分析出来的，不是随便配的。点了这道菜的顾客，87% 还点了这两道菜。服务员一提，顾客一般都会要。这就是数据驱动的加单。"

## 三商户差异演示要点

### czyz（尝在一起）—— 翻台率 → 桌台周转优化

**演示侧重点**：

- **Stage 1** 重点演示桌台拓扑图的实时状态（强调"翻台效率可视化"）
- **Stage 8** 重点展示翻台率指标趋势，对比行业均值
- **说词补充**：
  > "尝在一起的核心痛点是翻台率。高峰期桌台利用率不够高，等位顾客等太久就走了。屯象OS通过精准桌台调度，帮你把翻台率从 2.8 次/天提升到行业均值 3.5 次/天。以你们门店的座位数计算，这相当于每天多 XX 笔订单。"
- **数据准备**：czyz 历史翻台率数据 ≥7天，驾驶舱趋势图可见明显变化

### zqx（最黔线）—— 复购率 → 会员运营深度

**演示侧重点**：

- **Stage 5** 重点演示会员复购分析面板（本月/上月/行业均值三线对比）
- **Stage 3** 重点演示菜品推荐（基于历史复购数据的关联推荐）
- **说词补充**：
  > "最黔线有一批忠实顾客，但你们之前没有系统的会员运营工具。屯象OS的会员引擎能识别哪些顾客即将流失，提前推一条复购券就能把他们拉回来。我们测算，把复购率提升5个百分点，等于凭空多出来一批老顾客。"
- **数据准备**：zqx 会员数据 ≥50 条，含至少 2 个等级（普通/银卡），近30天复购记录

## 异常备选脚本

以下为演示中可能遇到的故障场景及对应应急处置方式。技术保障人员需熟记，演示员保持镇定，按脚本回应。

---

### 场景 A：打印机离线

**触发条件**：点击打印后无响应，状态显示打印机离线。

**技术处置**：
1. 检查打印机 IP 配置是否正确
2. 检查安卓 POS 与打印机是否同网段
3. 重启打印机电源，等待 30 秒重试

**演示员说词**：
> "大家看到这里有一个打印队列缓存机制。现在打印机因为网络原因暂时离线，但打印任务并没有丢——它已经进入了本地队列。一旦打印机恢复，所有队列任务会自动打印，不会漏单。"

`[操作]` 展示 POS 后台 → 打印队列页面，显示"待打印：1条，状态：等待连接"。

---

### 场景 B：接口响应超时（>3s）

**触发条件**：某个操作加载圈转很久，或出现 503/504 错误。

**技术处置**：
1. 检查云端服务健康状态（`curl /health`）
2. 如云端超时，切换 Mac mini 本地 API 模式（修改 API base URL 到 `mac-mini:8000`）

**演示员说词**：
> "这里大家看到我们的骨架屏设计——系统在等待数据时，界面不会空白，而是显示占位骨架，有良好的过渡体验。同时系统会自动重试，或切换到本地服务继续工作。"

`[展示]` 演示骨架屏加载状态（可在演示机上开启慢速网络模拟）。

---

### 场景 C：Mac mini 断网（云端同步中断）

**触发条件**：Mac mini 无法连接云端，Tailscale 显示离线。

**技术处置**：
1. 无需立即处理（本地服务继续工作）
2. sync-engine 自动进入队列缓存模式

**演示员说词**：
> "这正好演示我们的离线能力。Mac mini 现在无法连接云端，但门店业务完全不受影响——收银、打印、点餐全部照常。所有本地产生的数据都存在本地数据库里，一旦网络恢复，sync-engine 会自动把数据同步到云端，断网期间的数据一条不丢。"

`[操作]` 展示 Mac mini sync-engine 状态页面，显示"离线缓冲：X 条待同步，下次同步：网络恢复后自动执行"。

---

### 场景 D：门店网络完全断开（安卓 POS 也断网）

**触发条件**：门店路由器故障，所有设备断网。

**技术处置**：
1. 开启技术保障人员手机热点
2. 安卓 POS 连接热点继续工作

**演示员说词**：
> "极端情况下，即使整个门店断网，安卓 POS 主机有本地缓存模式。收银、开台、打印仍然可以操作，数据本地保存。顾客用现金付款时系统照样记账。网络恢复后数据自动上传，经营记录不会有空缺。这是对餐饮老板最重要的承诺：**数据不丢，生意不停**。"
