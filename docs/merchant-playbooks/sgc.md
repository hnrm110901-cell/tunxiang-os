## 一、商户简介

- **业态**：宴席 / 包间为主的高端餐饮
- **规模**：单店或少量门店，包间数多于散桌 — TBD（创始人补全）
- **现用 POS**：TBD（创始人确认。候选：天财商龙 / 客如云）
- **核心痛点**：宴席定金对账混乱（纸单+电话）；包间排期靠人盯；尾款追收拖延

## 二、DEMO 路径（10 步）

> 演示总时长 12-15 分钟。宴席全流程数字化是 sgc 主线。

| # | 操作 | 预期结果 |
|---|---|---|
| 1 | 打开 web-reception（前台接待，可在 iPad 上展示） | 包间排期日历显示当周宴席预订 |
| 2 | 新建宴席预订：客户「张总」，下周六 18:00，10 人圆桌 × 2 | 预订记录写入 tx-trade reservation 表，发出 RESERVATION.CREATED 事件 |
| 3 | 收取定金 ¥2000（微信支付） | tx-finance 入账：定金科目，关联宴席 ID；mv_daily_settlement 异步更新 |
| 4 | 切到驾驶舱"宴席健康度"看板 | 当周 8 场宴席，6 场已收定金，2 场待收，金额合计 ¥18,400 |
| 5 | 切到 web-pos，模拟宴席当天开宴：扫描包间 QR 码进入预订单 | 预订转开台，KDS 一键下发 8 道菜 |
| 6 | KDS（安卓平板）显示：菜品按"上菜节奏"分组（前 4 道凉菜 → 6 道热菜 → 汤 → 主食）| 出餐调度 Agent 边缘 Core ML 计算预计完餐时间 75 分钟 |
| 7 | 模拟尾款结算：扣减定金、补缴 ¥3,800 | tx-trade 发 ORDER.PAID（含定金抵扣明细），mv_store_pnl 更新 |
| 8 | 全电发票申请（金税四期） | nuonuo 适配器调用 → 返回发票编号（mock，不实发） |
| 9 | 触发日结 → 当日宴席收入合计、毛利率、成本占比 | mv_store_pnl 显示当日宴席净利率 32% |
| 10 | reset 脚本清演示数据 | 环境复位 |

## 四、演示前 60 秒话术（现场用）

> "尚宫厨做宴席和包间，这门生意最大的痛点不是出餐，是流程：定金谁收的、收了多少、退过多少、尾款补了没——这些数据如果用纸单+电话+微信群管理，一年下来对不上账的金额可能有六位数。
>
> 屯象OS把宴席全流程数字化：客户预订、定金收取、菜单确认、上菜节奏、尾款补缴、发票开具、月底对账，全部一条链路。每一笔钱、每一道菜、每一个时间点，老板手机上一眼能看到。
>
> 我们今天演示的不是软件功能，是经营透明度——三个月内，把你们对账时间从 3 天压到 30 分钟，定金回款率从 80% 提到 95%。"

## 五、回退方案

| 场景 | 紧急动作 |
|---|---|
| 包间日历加载慢 | 跳过第 1-2 步，直接从既有预订单进入演示 |
| 定金支付失败 | 切手动入账模式：tx-finance 后台直接录单 |
| KDS 上菜节奏算错 | `flags/agents/serve_dispatch` 关 → 改演示"手动派单"流程 |
| 全电发票超时 | 跳过第 8 步，话术："发票模块今天先不展开" |
| 日结对账不平 | 跳过第 9 步 |
| 整体炸了 | `bash scripts/demo-reset.sh && python3 scripts/seed_demo.py --merchant sgc --check` |

## 六、关键指标看板（演示中要展示的 3 个数字）

1. **本周宴席数 + 定金回款率** —— 8 场 / 75% 已收定金
2. **当日宴席净利率** —— 32%（mv_store_pnl 计算）
3. **平均尾款补缴时长** —— 宴席结束后 14 分钟内（vs 行业均值 48 小时）

## 七、已知红线

- **存酒押金多次续存**（高端宴席常见）：Tier 1 §19 二审中，**演示中不要打开存酒模块**

- **tiancai-shanglong 批量同步**：7 处 broad except，演示中不要触发"全量同步"按钮
- **yiding RLS**：跨租户隔离未做，**不要打开"全国预订"视图**
- **CRDT 4 小时断网**：不要现场拔网线
- **Master Agent 链式编排**：Wave 4 H2 在做，不展示
