# 尝在一起（czyz） DEMO Playbook

> Sprint F / F3 — 三商户演示脚本之一。
> 与 `docs/demo-playbook-store-fullflow.md` 配合，本文聚焦 czyz 专属切片。

---

## 一、商户简介

- **业态**：连锁中餐正餐
- **规模**：长沙起家，多店连锁，桌台数 / 翻台率为核心 KPI
- **现用 POS**：品智收银（pinzhi）—— 已与屯象适配器打通
- **核心痛点**：高峰期翻台率不够高，等位流失明显；折扣口子大，毛利失控；月底对账慢

## 二、DEMO 路径（10 步）

> 演示总时长 12-15 分钟。每步预期结果列在右侧。

| # | 操作 | 预期结果 |
|---|---|---|
| 1 | 打开安卓 POS（商米 T2），登录收银员账号 | WebView 加载 web-pos React App，显示桌台图，左侧 Agent 预警栏空 |
| 2 | 点 6 号桌（上一桌已结账，状态可清台） | 桌态切"清台中"→ 30 秒后自动转"待客" |
| 3 | 新开台：4 人桌，点单：2 主菜 + 1 凉菜 + 4 饮料 | 订单写入 tx-trade，本地 PG 同步，KDS（安卓平板）实时收到 |
| 4 | 收银台请求 8.8 折优惠 | **折扣守护 Agent** 弹窗：当前毛利率 22%，触发软告警（需店长二次确认） |
| 5 | 店长扫码二次确认 → 微信支付 | tx-trade 发出 ORDER.PAID + DISCOUNT.APPLIED 事件，金额单位"分" |
| 6 | 后厨 KDS 屏：菜品状态由"已下单"→"制作中"→"出餐" | 出餐时间 < 18 分钟（出餐调度 Agent 边缘 Core ML 预测线对比） |
| 7 | 切换到驾驶舱（web-admin），看翻台率趋势图 | 当日翻台率 3.4 次/桌（行业均值 3.5），趋势线连续 7 天可见 |
| 8 | 切到"折扣健康度"物化视图（mv_discount_health） | 8.8 折单当日命中 12 单，平均毛利 21.8%，无低于 15% 阈值告警 |
| 9 | 触发日结（tx-ops daily_settlement）| 发出 SETTLEMENT.DAILY_CLOSED 事件，前后厅对账一致 |
| 10 | 演示结束：用 reset 脚本清演示数据，恢复 seed | DEMO 环境复位，可重跑 |

## 四、演示前 60 秒话术（现场用）

> "尝在一起的核心生意是连锁正餐，翻台率是命脉。屯象OS不是叠加一层 AI，而是把你们现在用的 23 套系统替换成一套——POS、会员、KDS、报表都在里面。
>
> 今天演示重点三件事：第一，折扣不再失控，每笔单都有 Agent 实时校验毛利底线；第二，桌台调度透明化，你能看到每桌的翻台率趋势；第三，对账自动化，日结一键完成，发票走金税四期合规通道。
>
> 我们不卖软件功能，我们交付经营结果。三个月内把翻台率从 2.8 提到 3.5，每天多十单，按 80 元客单价算，月增收六位数——这个数字是我们和品智的真实数据测算出来的。"

## 五、回退方案

| 场景 | 紧急动作 |
|---|---|
| Agent 预警弹窗卡住 | `flags/agents/discount_guard` 关 → 重试该订单 |
| KDS 不收单 | 回到安卓 POS WebView，点重连；若仍然失败，切纸单兜底 |
| 适配器同步报错 | `python3 scripts/score_adapters.py --adapter pinzhi --json -` 查上次评分；停掉 sync-engine，手动跳过演示中"同步"步骤 |
| 日结对账不平 | 跳过第 9 步，话术过渡："日结模块今天先不展开，我们月底对账时再回访" |
| 整体环境炸了 | `bash scripts/demo-reset.sh && python3 scripts/seed_demo.py --merchant czyz --check` |

## 六、关键指标看板（演示中要展示的 3 个数字）

1. **当日翻台率** —— 目标 3.4+，近 7 天趋势线（驾驶舱主图）
2. **折扣健康度** —— 当日 12 笔折扣单，平均毛利 21.8%，零低于 15% 阈值
3. **出餐 P99** —— < 20 分钟（出餐调度 Agent 边缘预测）

> 三个数字必须现场可见，并且对得上后台数据库（mv_discount_health / 翻台率视图 / 出餐时间统计）。
