## 第一章 总则

### 第一条 目的

建立屯象OS软件开发全生命周期的安全管理规范，将安全要求内嵌到开发流程的每个阶段（安全左移），防止在代码层面引入安全漏洞，满足等保三级对"软件容错性"和"软件安全"控制点的要求。

### 第二条 依据

1. GB/T 22239-2019《信息安全技术 网络安全等级保护基本要求》第三级软件安全要求；
2. OWASP Top 10 Web应用安全风险（2021版）；
3. CWE/SANS Top 25 危险软件缺陷；
4. 本公司《信息安全管理办法》（TXK-SEC-001）；
5. 本公司CLAUDE.md项目宪法（开发规范核心文档）。

### 第三条 适用范围

本规定适用于屯象OS所有软件组件的开发活动，包括：
- 后端服务（FastAPI微服务体系，Python）；
- 前端应用（React Web App，TypeScript）；
- 安卓POS壳层（Kotlin）；
- iOS/iPad壳层（Swift）；
- 边缘服务（Mac mini Python服务 + Core ML桥接）；
- 数据库迁移脚本（Alembic）；
- 基础设施代码（Docker Compose、Shell脚本）。

## 第二章 安全开发生命周期（SDL）

### 第四条 需求阶段安全要求

**威胁建模（Threat Modeling）**：

新功能或重大功能变更在进入设计阶段前，须完成威胁建模分析，识别潜在安全风险。威胁建模使用STRIDE方法：

| 威胁类型 | 说明 | 屯象OS典型场景 |
|---------|------|-------------|
| S（欺骗/Spoofing）| 伪造身份 | 伪造租户身份访问他人数据 |
| T（篡改/Tampering）| 修改数据 | 篡改交易金额、折扣记录 |
| R（抵赖/Repudiation）| 否认操作 | 否认执行了高权限操作 |
| I（信息泄露/Information Disclosure）| 数据外泄 | 跨租户数据泄露、PII泄露 |
| D（拒绝服务/Denial of Service）| 服务中断 | API接口被滥用、数据库连接耗尽 |
| E（权限提升/Elevation of Privilege）| 越权操作 | 普通员工访问财务数据 |

涉及以下场景的功能须强制完成威胁建模：
- 新增对外API端点；
- 涉及PII数据处理的功能；
- 支付相关功能；
- 多租户权限模型变更；
- AI Agent决策逻辑（防止提示词注入、决策被操纵）。

### 第五条 设计阶段安全要求

**安全架构核心原则**（须在设计阶段落实）：

1. **多租户隔离设计**：
   - 所有涉及租户数据的API，必须在设计时明确RLS策略；
   - 严禁"查全量再过滤"模式，必须在数据库层面强制隔离；
   - 新增数据表必须包含`tenant_id`字段，并配置对应的RLS策略。

   ```sql
   -- 正确的RLS策略设计（使用JWT Claim）
   -- 须在设计评审时确认session变量的设置时机
   CREATE POLICY tenant_isolation ON orders
     USING (tenant_id = current_setting('app.current_tenant_id')::UUID);
   ```

2. **最小权限设计**：
   - API端点权限粒度到具体操作（read/write/admin）；
   - 数据库账号权限精确到表级别；
   - Agent决策权限须在设计时定义边界。

3. **数据流安全设计**：
   - 明确PII数据在系统中的流转路径；
   - 设计阶段确定脱敏规则和加密点；
   - 敏感数据不得出现在URL、GET参数、日志中。

### 第六条 开发阶段安全要求

**安全编码规范（引用CLAUDE.md核心规则）**：

#### 6.1 RLS多租户安全（最高优先级）

```python
# 正确：在请求生命周期开始时设置租户上下文
async def set_tenant_context(tenant_id: UUID, db: AsyncSession):
    """每个API请求必须调用此函数设置RLS上下文"""
    await db.execute(
        text("SELECT set_config('app.current_tenant_id', :tenant_id, true)"),
        {"tenant_id": str(tenant_id)}
    )

# 错误示例（严禁）：
# SELECT * FROM orders WHERE tenant_id = :id  -- 绕过RLS，应用层过滤
# 不设置RLS上下文直接查询                      -- 导致数据泄露
```

**等保三级关键提示**：RLS策略使用`current_setting('app.current_tenant_id')`时，必须确认该session变量在每次请求中被正确设置。根据内部安全审计发现，至少有3张表的RLS策略因session变量未正确设置而实际不生效（参见内部安全漏洞报告），这是当前最高优先级的安全修复项。

#### 6.2 SQL注入防护

```python
# 正确：参数化查询（SQLAlchemy ORM或参数化SQL）
result = await db.execute(
    select(Order).where(Order.order_id == order_id)
)

# 正确：使用参数化的原生SQL
result = await db.execute(
    text("SELECT * FROM orders WHERE order_id = :id"),
    {"id": order_id}
)

# 错误（严禁）：字符串拼接SQL
# query = f"SELECT * FROM orders WHERE order_id = '{order_id}'"  # ❌
```

#### 6.3 API认证与授权

```python
# 所有API端点须明确声明认证要求
from fastapi import Depends, HTTPException, status
from app.auth import require_auth, require_permission

@router.get("/orders/{order_id}")
async def get_order(
    order_id: UUID,
    current_user: User = Depends(require_auth),        # 强制认证
    _: None = Depends(require_permission("orders:read")),  # 强制权限
    db: AsyncSession = Depends(get_db),
):
    # 设置RLS上下文（必须在每个API处理函数开始时执行）
    await set_tenant_context(current_user.tenant_id, db)
    ...

# 错误（严禁）：无认证的敏感端点
# @router.get("/orders/{order_id}")  # ❌ 缺少认证依赖
```

#### 6.4 密钥和敏感信息处理

```python
import os
from app.config import get_secret

# 正确：从KMS或环境变量获取
JWT_SECRET = get_secret("JWT_SECRET_KEY")  # 通过KMS获取
DATABASE_URL = os.environ["DATABASE_URL"]  # 从环境变量获取

# 错误（严禁）：
# JWT_SECRET = "hardcoded_secret_key_12345"  # ❌ 硬编码
# JWT_SECRET = "my-jwt-secret"               # ❌ 弱密钥

# 日志中禁止输出敏感信息
import logging
logger = logging.getLogger(__name__)

# 正确：脱敏后记录
logger.info(f"User login: user_id={user.id}, phone={mask_phone(user.phone)}")

# 错误（严禁）：
# logger.info(f"User login: phone={user.phone}, password={password}")  # ❌
```

#### 6.5 输入验证与输出编码

```python
from pydantic import BaseModel, validator, constr

class OrderCreateRequest(BaseModel):
    """所有API输入须使用Pydantic模型强制验证"""
    table_number: int
    items: list[OrderItem]
    # 使用constr限制字符串长度，防止超长输入
    note: constr(max_length=500, strip_whitespace=True) = ""

    @validator('items')
    def validate_items_not_empty(cls, v):
        if not v:
            raise ValueError("订单至少需要一个商品")
        if len(v) > 100:
            raise ValueError("单笔订单商品数量不超过100")
        return v
```

#### 6.6 AI Agent安全编码规范

```python
# Agent决策须严格校验三条硬约束（参见CLAUDE.md第六章）
class AgentDecisionValidator:
    def validate(self, decision: AgentDecision) -> bool:
        """所有Agent决策必须通过三条硬约束校验"""
        # 1. 毛利底线校验（任何折扣不可使毛利低于阈值）
        if not self._check_gross_margin(decision):
            raise ConstraintViolation("毛利底线约束违规")
        # 2. 食安合规校验（临期/过期食材不得出品）
        if not self._check_food_safety(decision):
            raise ConstraintViolation("食品安全约束违规")
        # 3. 出餐时间校验（不超过门店设定上限）
        if not self._check_service_time(decision):
            raise ConstraintViolation("出餐时间约束违规")
        return True

# 防止提示词注入（用户输入不得直接拼接到系统提示词）
def build_agent_prompt(user_input: str, system_context: dict) -> str:
    # 用户输入须经过安全处理，不得直接拼接到系统提示词
    sanitized_input = sanitize_user_input(user_input)
    return SYSTEM_PROMPT_TEMPLATE.format(
        context=system_context,
        user_query=sanitized_input  # 严格隔离用户输入和系统指令
    )
```

### 第七条 测试阶段安全要求

**安全测试分类和要求**：

| 测试类型 | 说明 | 执行时机 | 工具 | 通过标准 |
|---------|------|---------|------|---------|
| SAST（静态应用安全测试）| 代码扫描 | 每次提交PR | bandit（Python）, ESLint安全规则 | 无HIGH/CRITICAL级别问题 |
| SCA（软件成分分析）| 依赖漏洞扫描 | 每次构建 | trivy, Safety | 无已知CRITICAL/HIGH CVE |
| DAST（动态应用安全测试）| 运行时扫描 | 版本发布前 | OWASP ZAP（自动化）| 无高危漏洞 |
| 渗透测试 | 手工/自动化攻击测试 | 每半年 | 专业工具或外包 | 无CVSS≥7.0的未修复漏洞 |
| 多租户隔离测试 | RLS策略验证 | 每次数据库迁移后 | 自动化测试用例 | 100%隔离测试通过 |
| API认证测试 | 未授权访问测试 | 每次API变更后 | 自动化测试 | 所有端点须认证 |

**多租户隔离自动化测试要求**：

```python
# 必须包含的隔离测试用例（关键等保合规测试）
class TestMultiTenantIsolation:
    async def test_tenant_a_cannot_read_tenant_b_orders(self):
        """租户A不能读取租户B的订单"""
        # 以租户A身份查询，验证不能获取租户B的数据
        orders = await order_service.list_orders(tenant_id=TENANT_A_ID)
        assert all(o.tenant_id == TENANT_A_ID for o in orders)
        assert not any(o.tenant_id == TENANT_B_ID for o in orders)

    async def test_rls_policy_active_without_context(self):
        """验证未设置RLS上下文时查询被拒绝"""
        # 不设置tenant上下文，应该返回空或报错
        with pytest.raises(Exception):
            await db.execute(text("SELECT * FROM orders"))
```

## 第三章 安全工具链

### 第九条 工具链配置要求

以下工具必须在所有开发环境中安装和配置，并集成到CI/CD管道：

**git-secrets（密钥泄露防护）**：
```bash
# 安装配置（所有开发环境必须执行）
brew install git-secrets  # macOS
git secrets --install     # 在仓库中安装钩子
git secrets --register-aws

# 添加屯象OS自定义密钥模式
git secrets --add 'sk-ant-[a-zA-Z0-9]{95}'        # Anthropic API Key
git secrets --add 'AKID[A-Z0-9]{32}'              # 腾讯云AccessKey
git secrets --add '[0-9]{18,19}\s*[0-9a-zA-Z]{32}'  # 腾讯云SecretKey
git secrets --add 'wx[a-f0-9]{16}'                # 微信AppID格式
git secrets --add '(password|passwd|secret|key)\s*=\s*["\'][\w\-]{8,}' # 通用密码
```

**bandit（Python SAST）**：
```bash
# 安装
pip install bandit

# 执行扫描（CI/CD中配置）
bandit -r services/ shared/ edge/ \
  -ll \              # 仅报告中危及以上
  -x tests/,venv/ \ # 排除测试目录
  -f json \          # JSON格式输出
  -o bandit-report.json

# 通过标准：无HIGH/CRITICAL级别的问题
# 如有MEDIUM级别问题，需要人工评审后方可合并
```

**trivy（依赖漏洞扫描）**：
```bash
# 扫描Python依赖
trivy fs --security-checks vuln \
  --severity CRITICAL,HIGH \
  --exit-code 1 \           # 发现问题时返回非零退出码（阻断CI）
  requirements.txt

# 扫描容器镜像
trivy image --severity CRITICAL,HIGH \
  --exit-code 1 \
  txk-api-service:latest

# 扫描整个代码目录
trivy fs --security-checks vuln,secret \
  --exit-code 1 \
  .
```

**CI/CD安全集成（GitHub Actions）**：
```yaml
# .github/workflows/security-check.yml
name: Security Scan

on: [push, pull_request]

jobs:
  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Run git-secrets
        run: |
          git secrets --list
          git secrets --scan

      - name: Run bandit (Python SAST)
        run: |
          pip install bandit
          bandit -r services/ shared/ edge/ -ll -x tests/ -f json -o bandit-report.json
          # 如果bandit返回非零退出码，CI失败
        continue-on-error: false

      - name: Run trivy (dependency scan)
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'
          scan-ref: '.'
          severity: 'CRITICAL,HIGH'
          exit-code: '1'

      - name: Upload security reports
        uses: actions/upload-artifact@v3
        if: always()
        with:
          name: security-reports
          path: |
            bandit-report.json
            trivy-report.json
```

## 第四章 第三方组件管理

### 第十条 第三方组件引入安全评估

在引入新的第三方组件（Python包、npm包、第三方服务）之前，须完成以下安全评估：

**评估检查清单**：

- [ ] 1. 组件来源可信（官方渠道，非fork或镜像）；
- [ ] 2. 检查PyPI/npm上的下载量和维护活跃度（近3个月有更新）；
- [ ] 3. trivy扫描无已知CRITICAL/HIGH漏洞；
- [ ] 4. 检查组件LICENSE兼容性（商业使用限制）；
- [ ] 5. 不引入不必要的权限（如访问文件系统、网络的组件）；
- [ ] 6. 版本锁定（requirements.txt/package-lock.json指定精确版本）。

### 第十一条 依赖版本管理

1. **版本锁定**：所有第三方依赖须锁定精确版本，禁止使用`*`或`>=`宽松版本约束（生产依赖）；
2. **依赖更新**：每月检查依赖更新情况，安全补丁版本须在7天内评估是否升级；
3. **高危CVE处理**：依赖组件发现CRITICAL/HIGH级别CVE，须在48小时内评估是否升级，若无法升级须制定临时缓解方案；
4. **Dependabot配置**：在GitHub仓库中配置Dependabot自动检测依赖漏洞，自动创建PR。

## 第五章 环境隔离要求

### 第十二条 三环境严格隔离

屯象OS维护三个隔离的运行环境，各环境数据和配置完全独立，禁止跨环境数据混用：

| 维度 | 开发环境（dev）| 测试/预发布环境（staging）| 生产环境（prod）|
|------|-------------|---------------------|--------------|
| 数据库 | 本地PostgreSQL + 脱敏样本数据 | 独立CDB实例 + 脱敏数据 | 腾讯云CDB广州（真实数据）|
| 密钥 | 本地.env.local（不提交Git）| 测试专用密钥 | 腾讯云KMS |
| API密钥 | 测试用沙箱密钥 | 测试用沙箱密钥 | 生产密钥（KMS）|
| 访问控制 | 本地，开放 | 受限，团队内部 | 严格，堡垒机+MFA |
| 真实PII数据 | 禁止 | 禁止（须脱敏）| 是，须加密保护 |

**禁止行为**：
- 禁止将生产数据（含真实PII）导出到开发或测试环境；
- 禁止在生产环境运行开发或调试代码；
- 禁止生产数据库凭证在开发环境中使用（即使只读）；
- 禁止跳过staging直接发布到生产（紧急热修复须额外审批）。

## 第六章 代码审查要求

### 第十三条 单人开发安全代码审查说明

**现阶段特殊说明**：屯象OS目前为单人开发模式。在正式外部代码评审条件成熟前，执行以下替代措施：

**AI辅助代码安全审查**：

每次重要功能合并前，通过Claude Code或类似工具进行安全重点审查，重点检查项目：
1. RLS策略设置是否正确（session变量是否在请求生命周期内被设置）；
2. 新增API端点是否有认证和权限检查；
3. 是否有SQL注入风险（字符串拼接SQL）；
4. 是否有明文密钥或敏感信息；
5. 输入验证是否完整（边界值、类型、长度）。

**审查记录要求**：
- 重要功能合并（新API、数据库Schema变更）须留存代码审查记录；
- 审查记录可以是自评注释、Claude对话记录截图、Pull Request评论；
- 审查记录与代码一同归档在Git历史中。

**当团队规模扩大时**：
当增加第二名开发人员时，须立即实施强制同行代码评审（Peer Review）要求，所有合并至main分支的代码须至少一名其他开发人员审查通过。
