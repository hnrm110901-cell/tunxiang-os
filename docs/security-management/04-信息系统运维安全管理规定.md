## 第一章 总则

### 第一条 目的

为规范屯象OS信息系统的日常运维操作行为，建立完整的变更管理流程和操作审计机制，解决运维操作日志记录不完整、变更缺乏管控等安全薄弱环节（参考等保测评典型扣分项），满足等保三级对"安全管理中心"和"运维管理"控制点的要求，特制定本规定。

> 编写背景说明：根据同类SaaS产品等保测评报告（以下简称"参考测评报告"）显示，"未详细记录运维操作日志"是导致扣分的高频问题。本规定针对性建立全流程操作留痕机制，确保运维操作可追溯、可审计。

### 第二条 依据

1. GB/T 22239-2019《信息安全技术 网络安全等级保护基本要求》第三级安全管理中心；
2. GB/T 28448-2019《信息安全技术 网络安全等级保护测评要求》；
3. 本公司《信息安全管理办法》（TXK-SEC-001）；
4. 本公司《网络安全管理规定》（TXK-SEC-002）。

### 第三条 适用范围

本规定适用于所有对屯象OS生产环境的运维操作，包括：
- 服务器操作系统层面的配置和维护；
- 数据库管理和维护操作；
- 应用服务的部署、重启和配置修改；
- 网络设备和安全设备的配置变更；
- 第三方人员参与的运维支持活动。

## 第二章 运维操作规范

### 第四条 堡垒机强制要求

所有对生产服务器的运维操作必须遵守以下规定：

1. **强制堡垒机接入**：所有服务器SSH连接必须通过JumpServer堡垒机，禁止任何形式的直接SSH访问。生产服务器安全组仅对堡垒机IP开放22端口；

2. **自动会话录屏**：JumpServer配置自动录制所有运维会话，包括：
   - 完整的命令执行记录（含输入输出）；
   - 文件上传/下载操作记录；
   - 会话开始/结束时间戳；
   - 操作账号标识。
   录屏文件保留180天，存储于COS加密归档；

3. **账号与人员对应**：堡垒机账号与运维人员一一对应，禁止账号共享。外包人员使用临时独立账号；

4. **MFA强制验证**：登录堡垒机须完成MFA验证（TOTP + 密码双因素）；

5. **会话超时设置**：堡垒机SSH会话闲置超过30分钟自动断开，需重新认证。

### 第五条 禁止行为清单

以下行为严格禁止，违规按《信息安全管理办法》第五章处置：

| 禁止行为 | 说明 |
|---------|------|
| 直接SSH访问生产服务器 | 绕过堡垒机的任何访问方式均属违规 |
| root账号日常运维 | 生产服务器禁止使用root账号执行日常操作，须使用有sudo权限的个人账号 |
| 明文存储凭证 | 禁止在服务器上以任何形式（文件、环境变量配置文件、脚本）明文存储密码、密钥 |
| 生产环境测试调试 | 禁止在生产环境进行功能测试、调试代码，须使用测试/预发布环境 |
| 未授权数据导出 | 禁止将生产数据库数据导出到非授权存储位置 |
| 绕过变更流程操作 | 禁止在未提交变更工单的情况下修改生产环境配置 |
| 跨租户数据访问 | 禁止在运维操作中访问非当前工单涉及的租户数据 |

### 第六条 敏感操作执行规范

以下操作属于高风险/高影响操作，须执行额外的安全控制：

**数据库直接操作**（高风险）：
1. 必须先在测试环境验证SQL语句；
2. 批量UPDATE/DELETE操作须先查询确认影响范围；
3. 操作前备份目标表数据（pg_dump单表）；
4. 所有DML操作在事务中执行，确认正确后再提交；
5. 操作时间须记录在变更工单中。

**服务停机操作**（高影响）：
1. 提前在系统公告中发布维护通知（重要变更提前24小时，紧急变更提前1小时）；
2. 准备回滚方案，确认可以快速恢复；
3. 停机窗口选择业务低峰期（优先选择凌晨02:00-06:00）。

**配置文件修改**（不可逆风险）：
1. 修改前备份原始配置文件；
2. 使用版本控制管理配置文件（修改后提交Git）；
3. 修改后立即验证服务功能正常。

## 第三章 变更管理流程

> 本章针对"参考测评报告"中"未详细记录运维操作日志、变更缺乏管控"问题建立完整解决方案。

### 第七条 变更分类

| 变更类型 | 描述 | 示例 |
|---------|------|------|
| 标准变更 | 低风险、有成熟操作规程的例行变更 | 依赖包更新、日志级别调整 |
| 普通变更 | 中等风险，需要评估但影响可控 | 功能版本发布、配置参数调整、数据库索引变更 |
| 重大变更 | 高风险或影响范围广，可能导致服务中断 | 数据库Schema变更、网络架构调整、大批量数据迁移 |
| 紧急变更 | 修复生产安全漏洞或严重故障，需快速执行 | 高危漏洞热修复、生产故障紧急恢复 |

### 第八条 变更申请流程

```
┌─────────────────────────────────────────────────────────────────┐
│                        变更管理流程                              │
└─────────────────────────────────────────────────────────────────┘

1. 变更申请（提交变更申请表）
   ↓
2. 风险评估（评估影响范围、回滚方案）
   ↓
3. 审批确认（李淳通过企业微信审批，留存截图）
   ↓
4. 变更实施（通过堡垒机执行，自动录屏）
   ↓
5. 变更验证（验证功能正常，监控指标无异常）
   ↓
6. 变更记录（更新工单状态，归档所有材料）
```

### 第九条 各类变更审批要求

| 变更类型 | 申请提前量 | 审批方式 | 可否周末执行 |
|---------|---------|---------|------------|
| 标准变更 | 提前4小时通知 | 企业微信文字确认 | 是 |
| 普通变更 | 提前24小时申请 | 企业微信文字确认 + 记录截图 | 非高峰期可以 |
| 重大变更 | 提前72小时申请 | 正式变更申请表 + 企业微信确认 | 否（除非紧急） |
| 紧急变更 | 事前口头确认 | 企业微信确认（事后补变更工单）| 是 |

所有审批均由李淳作为最终审批人，通过企业微信留痕作为正式审批记录。审批截图归入变更档案。

### 第十条 变更记录保留要求

1. 所有变更记录保留至少**1年**；
2. 变更记录包含：变更申请表、审批记录、操作记录（堡垒机日志/录屏链接）、验证结果；
3. 变更记录存储位置：企业微信群历史记录 + Git仓库docs/changelogs/目录；
4. 重大变更记录纳入年度安全审计报告。

## 第四章 系统上线前安全检查清单

### 第十一条 新功能/版本上线检查清单

每次生产发布前，须逐项完成以下20项安全检查，并由操作人员签确认：

**一、代码安全检查（6项）**

- [ ] 1. 代码已通过CI/CD管道，所有自动化测试通过
- [ ] 2. bandit安全扫描无高危问题（SEVERITY:HIGH/CRITICAL）
- [ ] 3. trivy依赖漏洞扫描无已知高危CVE（CRITICAL/HIGH级别）
- [ ] 4. git-secrets扫描确认无密钥/凭证泄露
- [ ] 5. 代码评审已完成（自评或AI辅助评审），无明显安全缺陷
- [ ] 6. 新增API端点已配置认证和授权检查

**二、数据库变更检查（4项）**

- [ ] 7. 数据库迁移脚本已在测试环境完整验证
- [ ] 8. RLS策略已验证（多租户隔离测试通过）
- [ ] 9. 无裸露的 `SET app.current_tenant_id` 调用，正确使用session变量
- [ ] 10. 新增敏感数据字段已配置适当加密

**三、配置与权限检查（4项）**

- [ ] 11. 新增环境变量已通过KMS或SSM配置，无明文硬编码
- [ ] 12. 新增对外API端口/路由已通过WAF规则测试
- [ ] 13. 新功能的最小权限已验证（不允许超权访问）
- [ ] 14. 日志输出已验证，不包含敏感信息（密码、Token、PII）

**四、业务与功能验证（3项）**

- [ ] 15. 核心业务流程（点餐/收银/结账）已完整测试
- [ ] 16. 多租户数据隔离测试：租户A无法访问租户B数据
- [ ] 17. 错误处理已测试，错误信息不泄露系统内部细节

**五、回滚准备（3项）**

- [ ] 18. 已准备回滚方案，可在30分钟内恢复到上一版本
- [ ] 19. 数据库迁移有对应的降级脚本（down revision）
- [ ] 20. 已告知影响的租户/用户（如有停机影响）

**确认签字**：操作人员 _______________ 日期 _______________ 企业微信通知截图已归档。

## 第五章 外包与第三方人员访问管理

> 本章直接解决"参考测评报告"中"外部人员未签保密协议"扣分项。

### 第十二条 第三方访问原则

1. **最小权限**：第三方人员仅授予完成特定任务所需的最小权限；
2. **时限访问**：临时账号须设置明确的有效期，最长不超过任务周期；
3. **全程监督**：第三方人员的所有系统操作须通过堡垒机录制，不得单独操作生产系统；
4. **保密协议前置**：所有第三方人员在获得任何访问权限前，须签署保密协议；
5. **及时回收**：访问任务结束后**24小时内**注销临时账号，回收所有访问权限。

### 第十三条 第三方访问流程

**申请阶段**：
1. 提交外部人员访问申请（见附件）；
2. 说明访问目的、所需权限范围、访问时间段；
3. 第三方人员签署《信息安全保密协议》；
4. 李淳审批，企业微信留存审批记录。

**实施阶段**：
1. 创建临时账号（格式：temp-[姓名缩写]-[YYYYMMDD]）；
2. 配置最小权限，通过堡垒机授予访问；
3. 向第三方人员说明操作规范和禁止行为；
4. 全程通过堡垒机录屏记录操作；
5. 屯象科技人员陪同或远程监控（高风险操作须全程陪同）。

**结束阶段**：
1. 任务完成后立即禁用临时账号；
2. 审查操作日志，确认无异常操作；
3. 24小时内彻底删除临时账号；
4. 归档访问记录（申请表+审批截图+操作日志+保密协议）。

### 第十四条 云服务商责任边界

屯象OS基础设施托管于腾讯云，安全责任按照"共担模型"划分：

| 安全层级 | 腾讯云负责 | 屯象科技负责 |
|---------|---------|------------|
| 物理安全 | 数据中心物理安全、硬件安全 | 无需负责 |
| 基础设施 | 虚拟化层、网络基础设施 | 无需负责 |
| 计算资源 | 服务器硬件和虚拟化 | CVM实例OS安全配置 |
| 网络安全 | VPC基础网络隔离 | 安全组规则、WAF配置 |
| 数据库 | CDB硬件高可用 | 数据库用户权限、Schema安全 |
| 数据安全 | 物理介质销毁 | 数据加密、备份、访问控制 |
| 身份认证 | 腾讯云账号MFA | 应用层用户认证、JWT管理 |
| 应用安全 | — | 全部由屯象科技负责 |
| 合规责任 | 腾讯云等保合规证明 | 屯象OS等保定级和合规 |
