## 第一章 总则

### 第一条 目的

为保障屯象OS SaaS平台的数据安全和业务连续性，防范数据丢失、系统故障、灾难性事件等风险，规范数据备份和恢复管理流程，满足等保三级对"数据备份恢复"控制点的要求，特制定本规定。

### 第二条 依据

1. GB/T 22239-2019《信息安全技术 网络安全等级保护基本要求》第三级"数据备份恢复"控制点；
2. 《数据安全法》第二十七条数据保护义务；
3. 本公司《信息安全管理办法》（TXK-SEC-001）；
4. 腾讯云数据库备份服务文档及最佳实践。

### 第三条 适用范围

本规定适用于屯象OS系统中所有需要备份的数据资产，包括业务数据库、审计日志、系统配置及代码资产。

### 第四条 恢复目标（SLA）

| 指标 | 目标值 | 说明 |
|------|-------|------|
| 恢复时间目标（RTO） | ≤ 4小时 | 从灾难发生到系统恢复正常服务所需最长时间 |
| 恢复点目标（RPO） | ≤ 2小时 | 允许丢失的最长数据时间窗口 |
| 备份成功率 | ≥ 99.9% | 每月备份任务成功率目标 |
| 恢复验证成功率 | 100% | 每月恢复演练必须成功 |

## 第二章 备份范围与分类

### 第五条 数据分类定义

**核心数据**（最高优先级，丢失不可接受）：
- 客户业务数据库：会员信息、交易流水、订单记录、员工信息；
- 财务结算数据：账单、分账记录、对账数据；
- 租户配置数据：门店信息、菜单、权限配置。

**配置数据**（高优先级，丢失需要重建成本）：
- 系统配置文件：Docker Compose、Nginx配置、环境变量模板；
- 数据库Schema：DDL脚本、Alembic迁移文件；
- 安全配置：WAF规则、安全组配置快照；
- 源代码：全量代码仓库（GitHub私有仓库为主存储）。

**日志数据**（中优先级，用于审计和故障排查）：
- 操作审计日志：用户操作记录、API访问日志；
- 运维操作日志：JumpServer操作录屏、命令执行记录；
- 系统运行日志：应用日志、错误日志、慢查询日志；
- 安全事件日志：WAF拦截日志、登录失败日志。

## 第三章 备份策略

### 第六条 备份策略总表

| 数据类型 | 备份方式 | 备份频率 | 保留期限 | 主存储位置 | 备份存储位置 | 加密要求 |
|---------|---------|---------|---------|-----------|------------|---------|
| 业务数据库（PostgreSQL） | 全量 + WAL增量 | 全量：每日 02:00 / 增量：每2小时 | 30天（全量）/ 7天（增量）| 腾讯云CDB广州 | 腾讯云COS上海 | AES-256（CDB自动加密）|
| Redis缓存 | 全量RDB + AOF | 全量：每日 / AOF：实时 | 7天 | 腾讯云Redis广州 | — | TLS传输加密 |
| 审计日志 | 增量（流式） | 实时写入CLS | 180天 | 腾讯云CLS广州 | 腾讯云COS上海（归档）| 默认加密 |
| 运维录屏 | 全量文件 | 实时生成 | 180天 | JumpServer本地（热）| 腾讯云COS（冷归档）| AES-256 |
| 系统配置 | 全量 | 每次变更后触发 | 永久 | GitHub私有仓库 | 腾讯云COS广州 | TLS传输 |
| 源代码 | Git全量 | 每次提交 | 永久 | GitHub私有仓库 | — | GitHub加密存储 |
| 数据库DDL | 全量脚本 | 每次Schema变更后 | 永久 | GitHub私有仓库 | 腾讯云COS广州 | TLS传输 |
| 安全配置快照 | 全量 | 每周 + 每次变更后 | 1年 | 腾讯云COS广州 | — | 服务端加密 |

### 第七条 数据库备份详细规范

**PostgreSQL业务数据库备份**：

1. **每日全量备份**：
   - 备份时间：每日凌晨02:00（业务低峰期）；
   - 备份方式：pg_dump导出 + 腾讯云CDB快照双重保障；
   - 备份格式：自定义压缩格式（pg_dump -Fc）；
   - 备份文件命名：`txk-db-full-YYYYMMDD-HHMMSS.dump`；
   - 备份完成后自动校验文件完整性（SHA256校验和）。

2. **增量备份（WAL归档）**：
   - 频率：每2小时归档一次WAL日志；
   - WAL归档目标：腾讯云COS（广州），同步复制到上海；
   - 保留策略：7天内WAL日志完整保留，支持时间点恢复（PITR）。

3. **跨地域复制**：
   - 每日全量备份在完成后2小时内同步到上海地域COS；
   - 同步完成后发送告警通知确认；
   - 同步失败时，立即告警并在4小时内处理。

### 第八条 审计日志备份规范

1. **实时流式写入**：审计日志实时写入腾讯云CLS（日志服务），确保不丢失；
2. **不可删除策略**：CLS日志索引设置为只读，禁止删除；
3. **冷备归档**：超过30天的审计日志自动归档至腾讯云COS（标准-智能分层存储），保留至180天；
4. **完整性保障**：CLS日志支持哈希校验，防止篡改；
5. **合规保留**：180天内的审计日志须随时可查询，180天后可归档但须在24小时内恢复可查询状态。

## 第四章 备份验证

### 第九条 备份完整性验证

每次备份完成后，系统自动执行以下验证：

1. **文件完整性验证**：校验备份文件SHA256哈希值，与生成时记录的哈希值对比；
2. **备份大小验证**：备份文件大小须在前7次备份平均值的50%-200%范围内，超出范围触发告警；
3. **备份可读性验证**：每日全量备份完成后，自动在隔离的测试实例中执行`pg_restore --list`验证备份文件可读；
4. **验证结果记录**：验证结果写入监控日志，异常情况立即告警。

### 第十条 月度恢复演练要求

每月至少执行一次完整的数据库恢复演练，演练内容包括：

**演练步骤**：

1. 选取最近一次每日全量备份文件；
2. 在腾讯云临时测试实例中执行恢复；
3. 验证恢复后数据库的完整性（记录数、表结构、约束）；
4. 测试应用程序连接恢复后的数据库，验证基本业务功能可用；
5. 记录恢复用时（应满足RTO≤4小时目标）；
6. 演练完成后销毁临时测试实例；
7. 填写《备份恢复演练记录表》（见附件）。

**演练要求**：
- 演练时间：每月最后一个工作日的非业务高峰期；
- 演练记录须保留至少2年；
- 演练失败须立即分析原因，7天内完成整改并重新演练；
- 演练结果纳入年度安全审计报告。

## 第五章 恢复流程

### 第十一条 数据恢复申请与审批

任何生产数据的恢复操作须经过以下审批流程：

1. **提交恢复申请**：通过企业微信提交恢复申请，说明恢复原因、数据范围、目标时间点；
2. **评估影响**：评估恢复操作对当前业务的影响（是否需要停服、影响哪些租户）；
3. **审批确认**：由李淳作为最终审批人确认，通过企业微信留痕作为审批记录；
4. **制定恢复方案**：确定恢复方式（全量恢复 / 时间点恢复 / 部分表恢复）；
5. **执行恢复**：在堡垒机上执行恢复操作，全程录屏；
6. **恢复验证**：验证数据完整性，测试业务功能正常；
7. **归档记录**：填写恢复操作记录，归档保存。

### 第十二条 不同场景恢复方案

**场景一：单表数据误删/误改**
- 恢复方式：基于WAL日志的时间点恢复（PITR）或从全量备份中提取单表数据；
- 预计用时：1-2小时；
- 是否停服：通常不需要，可在备用实例恢复后迁移数据。

**场景二：数据库全库损坏**
- 恢复方式：从最近的每日全量备份恢复，叠加WAL增量日志；
- 预计用时：2-4小时（满足RTO≤4小时）；
- 是否停服：需要，恢复期间切换到维护页面。

**场景三：整个服务器灾难**
- 恢复方式：从腾讯云COS（上海地域）获取最近备份，在新实例恢复；
- 预计用时：3-4小时；
- 是否停服：需要，启动灾难恢复预案。

**场景四：勒索病毒加密数据**
- 恢复方式：立即隔离受感染服务器，从离线备份恢复（COS备份未被勒索病毒触及）；
- 预计用时：4小时内；
- 是否停服：需要全面停服，优先恢复核心业务。

### 第十三条 恢复操作技术规范

```bash
# 时间点恢复示例（操作须在堡垒机执行，须提前申请变更工单）

# 1. 从COS下载备份文件
coscmd download txk-db-full-YYYYMMDD-HHMMSS.dump /tmp/restore/

# 2. 验证备份文件完整性
sha256sum /tmp/restore/txk-db-full-YYYYMMDD-HHMMSS.dump
# 对比存储在安全位置的预期哈希值

# 3. 在测试实例恢复
pg_restore -h [临时实例地址] -U postgres -d tunxiang_restore \
  --jobs=4 /tmp/restore/txk-db-full-YYYYMMDD-HHMMSS.dump

# 4. 应用WAL日志（时间点恢复）
# 需配置recovery.conf指定目标时间点
# target_time = 'YYYY-MM-DD HH:MM:SS'

# 5. 验证记录数
psql -h [临时实例地址] -c "SELECT count(*) FROM orders;"
psql -h [临时实例地址] -c "SELECT count(*) FROM members;"

# 注意：生产环境恢复须额外验证RLS策略完整性
```
