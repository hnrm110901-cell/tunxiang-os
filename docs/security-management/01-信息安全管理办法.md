## 第一章 总则

### 第一条 目的

为规范屯象科技（以下简称"公司"）信息安全管理工作，保障屯象OS餐饮AI操作系统（以下简称"屯象OS"）及其承载的客户数据、业务数据的保密性、完整性和可用性，防范信息安全风险，满足国家网络安全等级保护第三级（以下简称"等保三级"）合规要求，特制定本办法。

### 第二条 依据

本办法依据以下法律法规及标准制定：

1. 《中华人民共和国网络安全法》（2017年6月1日施行）
2. 《中华人民共和国数据安全法》（2021年9月1日施行）
3. 《中华人民共和国个人信息保护法》（2021年11月1日施行）
4. 《网络安全等级保护条例》（征求意见稿）
5. GB/T 22239-2019《信息安全技术 网络安全等级保护基本要求》
6. GB/T 25070-2019《信息安全技术 网络安全等级保护安全设计技术要求》
7. GB/T 28448-2019《信息安全技术 网络安全等级保护测评要求》
8. 《个人信息安全规范》GB/T 35273-2020

### 第三条 适用范围

本办法适用于：

1. 公司全体成员（含全职、兼职、外包人员及实习生）；
2. 公司所有信息系统、网络设施、数据资产；
3. 与公司存在数据交换或系统接入关系的第三方供应商及合作伙伴；
4. 屯象OS SaaS平台的运营、开发、运维全过程。

### 第四条 术语定义

| 术语 | 定义 |
|------|------|
| 信息资产 | 对公司具有价值的信息及其处理、存储、传输所依赖的一切资源，包括数据、软件、硬件、服务、人员等 |
| 信息安全事件 | 可能导致信息资产损失、泄露、篡改、中断的已发生或潜在事件 |
| 多租户隔离 | 屯象OS通过PostgreSQL行级安全（RLS）策略实现的租户数据逻辑隔离机制 |
| 三权分立 | 系统管理员、审计管理员、安全管理员三个角色职责分离，相互制衡 |
| 等保三级 | 国家网络安全等级保护第三级，定级结论S3A2（业务信息安全等级三级、系统服务安全等级二级） |
| PII | 个人可识别信息（Personally Identifiable Information），包括姓名、手机号、身份证号、生物特征等 |
| RLS | 行级安全策略（Row-Level Security），PostgreSQL数据库层面的多租户数据隔离技术 |
| KMS | 密钥管理服务（Key Management Service），腾讯云密钥管理服务 |
| JWT | JSON Web Token，用于API身份验证的令牌机制 |

## 第二章 信息安全目标

### 第五条 保密性目标（Confidentiality）

确保信息只被授权人员访问，防止未授权的信息披露。具体目标包括：

1. **数据访问控制**：所有敏感数据访问须经身份认证和权限验证，数据库层面通过RLS策略强制执行租户隔离；
2. **传输加密**：所有网络传输使用TLS 1.2及以上协议，禁止明文传输敏感信息；
3. **存储加密**：机密级数据静态存储使用AES-256加密，API密钥及系统密钥由腾讯云KMS统一管理；
4. **PII保护**：客户手机号、身份证号等个人信息在存储和展示时执行脱敏处理，仅在业务必要时还原；
5. **最小权限原则**：每个账号仅授予其角色所需的最小操作权限，定期审查并回收多余权限。

### 第六条 完整性目标（Integrity）

确保信息在存储、处理和传输过程中保持准确、完整，防止未授权篡改。具体目标包括：

1. **数据校验**：交易流水、财务结算等关键数据使用HMAC算法进行完整性校验；
2. **审计追踪**：所有敏感操作产生不可篡改的审计日志，审计日志使用Append-Only模式存储；
3. **变更管理**：系统配置变更须经审批流程，变更前后状态均留存记录；
4. **代码完整性**：所有代码通过Git版本控制，生产环境部署须经CI/CD管道验证；
5. **数据备份验证**：备份数据定期进行完整性校验和恢复演练，确保备份可用。

### 第七条 可用性目标（Availability）

确保授权用户在需要时能够访问和使用信息系统。具体目标包括：

1. **系统可用性**：屯象OS核心服务年度可用性目标不低于99.5%；
2. **恢复时间目标（RTO）**：发生重大故障时，系统恢复时间不超过4小时；
3. **恢复点目标（RPO）**：数据恢复点目标不超过2小时，即最多丢失2小时内的数据；
4. **容灾备份**：关键数据在腾讯云广州和上海双地域备份，防止单点灾难；
5. **DDoS防护**：通过腾讯云Anti-DDoS服务防范大规模分布式拒绝服务攻击。

## 第三章 岗位职责

### 第八条 三权分立机制说明

为满足等保三级对"管理人员最小授权"和"权限分离"的要求，公司设立系统管理员、审计管理员、安全管理员三个安全角色，各角色职责严格分离、相互制衡。

**现阶段特殊说明**：公司处于初创期，三个安全角色暂由创始人李淳统一担任。为保障技术层面的权限分离，三个角色配置独立操作账号：

| 角色 | 账号标识 | 技术隔离措施 |
|------|---------|------------|
| 系统管理员 | sysadmin@tunxiang.com | 拥有系统配置权限，无审计日志查看权限 |
| 审计管理员 | auditor@tunxiang.com | 仅有审计日志只读权限，无系统配置权限 |
| 安全管理员 | secadmin@tunxiang.com | 拥有安全策略管理权限，无业务数据操作权限 |

所有跨角色操作须切换账号并留存操作记录，重要决策通过企业微信留痕作为审批记录。

### 第九条 系统管理员职责

**职责范围**：

1. 负责服务器、网络设备、数据库等基础设施的配置和维护；
2. 管理系统用户账号的创建、修改、禁用和删除；
3. 负责系统补丁和安全更新的测试与部署；
4. 管理系统备份配置和验证备份完整性；
5. 维护系统运行状态监控告警配置；
6. 处理系统故障和性能问题；
7. 管理JumpServer堡垒机账号和权限配置。

**禁止事项**：

1. **禁止**查看、修改或删除审计日志；
2. **禁止**修改安全策略配置（如防火墙规则、WAF策略）；
3. **禁止**访问非授权的业务数据；
4. **禁止**在未申请变更工单的情况下进行生产环境配置变更；
5. **禁止**使用root账号进行日常运维操作。

### 第十条 审计管理员职责

**职责范围**：

1. 定期审查系统操作日志、访问日志和安全事件日志；
2. 对审计日志进行分析，识别异常行为并及时上报；
3. 确保审计日志的完整性和不可篡改性；
4. 编制月度、季度、年度安全审计报告；
5. 对安全事件响应过程进行事后审计；
6. 核查用户权限配置是否符合最小权限原则；
7. 监督等保合规要求的执行情况。

**禁止事项**：

1. **禁止**修改或删除任何审计日志记录；
2. **禁止**修改系统配置或网络配置；
3. **禁止**操作业务数据（增删改）；
4. **禁止**向未授权人员披露审计日志内容；
5. **禁止**擅自对外发布审计报告。

### 第十一条 安全管理员职责

**职责范围**：

1. 制定和维护公司信息安全策略、规范和制度；
2. 管理防火墙、WAF、入侵检测等安全设备的策略配置；
3. 组织定期漏洞扫描和渗透测试，跟踪修复进展；
4. 管理API密钥、证书、JWT密钥等敏感凭证的生命周期；
5. 负责安全事件的研判、响应协调和应急处置；
6. 对第三方供应商进行安全评估；
7. 组织安全培训和安全意识教育；
8. 跟踪国内外安全威胁情报，评估对屯象OS的影响。

**禁止事项**：

1. **禁止**操作业务数据（增删改查敏感数据）；
2. **禁止**在未经审批的情况下开放新的网络端口或服务；
3. **禁止**查看或修改审计日志；
4. **禁止**将安全漏洞信息向未授权人员披露；
5. **禁止**私自与外部安全机构共享系统架构信息。

## 第四章 信息安全管理要求

### 第十二条 核心管理要求

以下10条核心要求适用于公司全体成员：

**要求一：身份认证要求**

所有信息系统必须实施强身份认证。生产系统、管理后台、堡垒机须启用多因素认证（MFA）。禁止使用弱密码、空密码或默认密码。密码复杂度、有效期等要求见《密码管理制度》。

**要求二：最小权限要求**

账号权限遵循最小权限原则，仅授予完成工作所需的最低权限。禁止权限滥用，禁止账号共享。员工离职或角色变更时，24小时内完成权限回收或调整。

**要求三：数据保护要求**

机密级数据（含客户PII、交易数据、财务数据）必须加密存储和传输。数据访问须留存访问日志。禁止将生产数据导出到非授权环境（如本地开发环境）。测试环境使用脱敏数据。

**要求四：网络安全要求**

生产网络实施分区隔离（防御区/服务区/管理区）。所有外网访问须经防火墙策略控制。服务器运维须通过JumpServer堡垒机，禁止直接SSH访问。远程办公须使用VPN或堡垒机。

**要求五：变更管理要求**

生产环境所有变更（代码发布、配置修改、网络调整等）须事先申请变更工单，经审批后方可执行，变更完成后留存记录。紧急变更须事后补充工单，保留记录。

**要求六：安全开发要求**

代码开发遵循安全编码规范（见《软件开发安全管理规定》）。所有代码须经安全扫描工具检查方可提交。禁止将密钥、密码、Token等敏感信息硬编码或提交至代码仓库。

**要求七：供应商管理要求**

采购或使用第三方服务前，须完成安全评估。第三方人员访问公司系统前，须签署保密协议。第三方访问全程监督，访问结束后24小时内注销临时账号。

**要求八：审计日志要求**

所有安全相关操作（用户登录、权限变更、数据访问、系统配置修改等）必须产生审计日志。审计日志须保留不少于180天，且不可被普通用户删除或篡改。

**要求九：备份与恢复要求**

关键数据按规定频率进行备份，备份数据异地存储。每月至少进行一次备份恢复演练，并记录演练结果。灾难恢复能力须满足RTO≤4小时、RPO≤2小时的目标。

**要求十：事件响应要求**

发现安全事件（含疑似事件）须在30分钟内上报安全管理员。安全管理员根据事件等级启动相应应急响应流程。安全事件处置完成后，须在48小时内提交书面事件报告。

## 第五章 违规处置

### 第十三条 违规等级划分

根据违规行为的严重程度，分为以下四个处置等级：

**一级：警告**

适用情形：
- 首次违反密码复杂度要求，经提醒仍未在规定时间内修改；
- 未经申请使用个人设备处理工作数据，无实质数据泄露风险；
- 轻微的操作规范违规，如未记录运维操作日志；
- 其他首次发生、危害轻微的违规行为。

处置措施：书面警告，要求在规定期限内整改，记入个人档案。

**二级：警告+整改**

适用情形：
- 重复出现一级违规行为（同一问题3个月内第二次发生）；
- 擅自开放未经授权的网络端口或服务；
- 在代码仓库中提交了敏感凭证信息（事后发现并立即清理）；
- 未经审批对生产环境进行变更操作，但未造成实际损失；
- 向未授权人员披露内部信息，但尚未导致数据外泄。

处置措施：书面警告+限期整改，视情况暂停相关系统访问权限，直至整改完成。

**三级：停职调查**

适用情形：
- 违规导致信息安全事件（含数据泄露、系统入侵等），损失尚在可控范围内；
- 故意规避安全控制措施；
- 向竞争对手或未授权第三方泄露公司机密信息；
- 伪造、篡改或删除审计日志；
- 明知系统存在高危漏洞而不上报。

处置措施：停止当前工作，启动内部调查，保全相关证据，视调查结果决定后续处置措施。

**四级：解除合作**

适用情形：
- 故意造成重大数据泄露事件（泄露客户PII超过1000条）；
- 恶意破坏公司信息系统或数据；
- 将公司系统访问权限出售或转让给第三方；
- 三级违规经调查情节严重，或停职调查期间继续违规；
- 触犯刑法的信息安全违规行为。

处置措施：立即解除劳动合同或合作协议，回收全部访问权限，依法追究法律责任，情节严重者向公安机关报案。

### 第十四条 违规处置程序

1. **发现与记录**：发现违规行为时，第一发现人须立即记录违规事实，包括时间、地点、行为描述和证据；
2. **初步研判**：安全管理员在12小时内完成违规等级初步研判；
3. **审批确认**：由李淳作为最终审批人确认处置等级，通过企业微信留痕作为审批记录；
4. **处置执行**：按对应等级执行处置措施，同步通知相关部门；
5. **归档记录**：所有违规处置记录存入安全事件档案，保留至少3年。

## 第六章 制度生效与修订

### 第十五条 生效日期

本办法自2026年4月1日起正式施行，由李淳签发生效。

### 第十六条 解释权

本办法的解释权归屯象科技所有，由安全管理员负责具体解释。

### 第十七条 修订机制

1. 本办法每年至少审查修订一次；
2. 以下情形应触发临时修订：
   - 相关法律法规发生重大变化；
   - 公司业务模式或技术架构发生重大调整；
   - 发生重大安全事件且本办法存在明显缺陷；
   - 等保测评发现本办法不满足要求。
3. 修订须经李淳审批，修订记录在文档末尾修订历史表中更新。
