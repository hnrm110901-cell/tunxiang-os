## 一、徐记海鲜业务复杂度画像

### 1.1 菜品体系（5层复杂度）

```
单品菜品
├── 普通单品（固定定价）
├── 活鲜单品（称重/条/头 计价）
│   ├── 称重菜：石斑鱼 68元/斤，按称重结果结算
│   ├── 计头菜：帝王蟹 980元/头，每头一价
│   └── 活率补偿：含死亡率损耗，价格含成本保底
├── 套餐
│   ├── 固定套餐：确定菜品+数量，一口价
│   └── 自选套餐（N选M）：每组M选N的灵活组合
└── 宴席菜单
    ├── 标准宴（188元/位，12菜）
    ├── 精品宴（298元/位，15菜）
    ├── 豪华宴（398元/位，18菜）
    └── 定制宴（协议定价）
```

### 1.2 渠道体系（6个渠道独立定价）

| 渠道 | 菜单 | 定价策略 | 特殊规则 |
|------|------|---------|---------|
| 堂食POS | 完整菜单 | 标准定价 | 活鲜称重 |
| 宴席 | 宴席专属菜单 | 套餐定价（元/位）| 同步出品 |
| 美团外卖 | 精选50%菜品 | 外卖溢价+15% | 无活鲜 |
| 饿了么 | 精选50%菜品 | 外卖溢价+15% | 无活鲜 |
| 抖音来客 | 套餐+团购 | 团购价 | 核销码 |
| 微信小程序 | 预订+自助点餐 | 标准定价 | 无活鲜 |

### 1.3 打单体系（8种单据类型）

| 单据 | 触发时机 | 打印机 | 特殊信息 |
|------|---------|--------|---------|
| 收银小票 | 结账成功 | 收银台 | 金额/支付方式/会员积分 |
| 厨打单（分单） | 订单确认后 | 各档口 | 菜品+桌号+做法+备注 |
| 活鲜称重单 | 称重确认后 | 活鲜档口 | 品种/重量/单价/金额 |
| 宴席通知单 | 宴席确认 | 厨师长打印机 | 菜单/开席时间/桌数/人数 |
| 催菜单 | 服务员催菜 | 对应档口 | 催菜菜品+催菜时间+桌号 |
| 重做单 | 菜品重做 | 对应档口 | 重做原因+优先级 |
| 补打单 | 手动补打 | 指定打印机 | 与原单相同+【补打】标识 |
| 企业挂账单 | 挂账结算 | 收银台 | 挂账单位+签字确认 |

### 1.4 KDS后厨管理（5个专属场景）

| 场景 | 档口 | 特殊需求 |
|------|------|---------|
| 堂食出餐 | 凉菜/热菜/海鲜/点心/主食 | 普通顺序出餐 |
| 宴席出餐 | 全部档口 | **同步出品倒计时**（所有桌同时开席）|
| 活鲜处理 | 海鲜档口 | 称重→确认→制作流程 |
| 外卖出品 | 独立外卖档口 | 超时预警（平台SLA） |
| 快餐叫号 | 快餐档口 | 顾客取餐叫号屏 |

## 四、技术实现细节

### 4.1 活鲜菜品数据模型

```python
# 新增字段（迁移到 dish 表）
pricing_method: str  # 'fixed'=固定价 / 'weight'=称重 / 'count'=条头
weight_unit: str     # 'jin'=斤 / 'liang'=两 / 'kg'=千克 / 'g'=克
price_per_unit: int  # 单位价格（分）—— weight时=每斤价，count时=每条/头价
min_order_qty: Decimal  # 最小点单量（活鲜常见0.5斤起）
tank_zone: str       # 鱼缸区域，如"A区3号缸"
tank_capacity: int   # 缸容量（条）
alive_rate_pct: int  # 历史成活率%（用于成本核算）
display_unit: str    # 展示单位，如"斤"/"条"/"头"/"位"
```

### 4.2 套餐N选M数据模型

```python
# combo_groups 表（新建）
class ComboGroup:
    combo_id: UUID      # 关联套餐
    group_name: str     # 组名，如"主菜（任选2款）"
    min_select: int     # 最少选 N
    max_select: int     # 最多选 M
    is_required: bool   # 是否必选
    sort_order: int

# combo_group_items 表（新建）
class ComboGroupItem:
    group_id: UUID
    dish_id: UUID
    extra_price_fen: int = 0  # 额外加价（有些高档菜需加价）
    is_default: bool    # 是否默认选中（固定套餐兼容）
```

### 4.3 宴席菜单数据模型

```python
# banquet_menus 表（新建）
class BanquetMenu:
    name: str               # "精品宴298元/位"
    tier: str               # standard/premium/luxury/custom
    per_person_fen: int     # 人均价格
    min_persons: int        # 最少人数
    min_tables: int         # 最少桌数
    is_active: bool

# banquet_menu_sections 表（新建）
class BanquetMenuSection:
    menu_id: UUID
    section_name: str       # "海鲜/凉菜/热菜/汤/主食/甜品"
    sort_order: int
    serve_sequence: int     # 出品顺序（凉菜=1，汤=5，主食=6）

# banquet_menu_items 表（新建）
class BanquetMenuItem:
    section_id: UUID
    dish_id: UUID
    quantity_per_table: int  # 每桌数量
    is_mandatory: bool       # 固定菜/选换菜
    alternatives: list[UUID] # 可替换菜品列表（宴席换菜）
    note: str
```

### 4.4 宴席同步出品控制

```python
# banquet_sessions 表（新建）
class BanquetSession:
    contract_id: UUID       # 关联宴席合同
    store_id: UUID
    scheduled_at: datetime  # 计划开席时间
    actual_open_at: datetime
    guest_count: int
    table_ids: list[UUID]   # 本次使用桌台

```

## 六、风险与注意事项

1. **活鲜称重精度**：称重数据从安卓POS秤通过JS Bridge传入，需做精度四舍五入（保留2位小数）和防抖处理
2. **宴席同步出品**：WebSocket推送延迟可能导致各档口不完全同步，需要引入Redis事务和确认机制
3. **渠道价格冲突**：多渠道价格独立时需要有价格变更的广播通知机制，防止渠道价格不一致
4. **活鲜库存**：活鲜按条/头管理，与克重换算需要配置系数，防止超卖
5. **打印机离线**：厨打单发送失败需要本地队列 + 重试，保证出品不遗漏
