# 品智极智版 × 屯象OS 功能对标与借鉴分析

> 基于品智极智版连锁餐饮运营系统操作说明（242页）深度学习

---

## 一、品智极智版功能全景（5 大模块）

| 模块 | 子功能 | 品智特色 |
|------|------|--------|
| **一、组织架构** | 企业信息、品牌管理、区域管理、门店管理（类别/标签/信息）、出品部门、用户管理（角色/用户） | 多品牌+多区域+门店类别标签+出品部门 |
| **二、支付管理** | 支付方式（13类别）、门店支付方式、支付方式互斥、财务类、优惠券 | 支付闭环 vs 记录类分离、沉睡天数、实收属性 |
| **三、菜单中心** | 菜品分类、菜品属性、菜品管理（普通/套餐/配料）、菜品发布、门店菜品、价格调整 | 品牌级→门店级三级菜品发布体系 |
| **四、门店运营** | 桌台管理（类别/营业区/桌台）、设备管理（打印）、KDS 管理（设置/上菜优先级）、单据管理（模板/排序）、单子条码、销售计划 | 出品部门+营业区+KDS 深度联动 |
| **五、营销管理** | 营销方案（特价/买赠/加价换购/再买/会员/订单折扣/满减）、执行顺序、共享互斥、三方优惠、会员消费奖励 | 7 种营销方案类型+共享互斥规则 |

## 二、值得借鉴的核心设计（12 项）

### 🔴 P0 — 必须借鉴（直接影响客户切换意愿）

#### 1. 快速开店（门店档案一键复制）
**品智做法**：选择新门店+老门店 → 勾选要复制的配置项（门店信息/支付方式/营销方案/桌台等）→ 一键复制完成。
**屯象OS 现状**：有 `new_store_setup.sh` 脚本但仅处理基础设施，没有业务配置复制。
**借鉴建议**：在 tx-org 或 gateway 中实现 **门店配置克隆 API**，支持选择性复制：菜品+支付+桌台+营销+KDS 配置。

#### 2. 菜品三级发布体系（品牌→门店→价格调整）
**品智做法**：品牌菜品库 → 菜品发布方案（批量发布到门店）→ 门店菜品（可微调）→ 价格调整方案（时段/节假日/外卖差异化定价）。
**屯象OS 现状**：Ontology 有 DishMaster→BrandMenu→StoreMenu 三级模型（V2.x），但 V3.0 tx-menu API 尚未实现发布流程。
**借鉴建议**：tx-menu 增加 `/api/v1/menu/publish-plans` 批量发布 + `/api/v1/menu/price-adjustments` 时段定价。

#### 3. 出品部门与 KDS/打印机的三方关联
**品智做法**：菜品 → 出品部门 → 打印机/KDS 自动关联。修改菜品出品部门时自动更新打印和 KDS 路由。
**屯象OS 现状**：web-kds 有三列看板，但缺少出品部门→打印机→KDS 的配置管理。
**借鉴建议**：tx-trade 增加出品部门模型 + 打印/KDS 路由配置，实现"加菜自动分单到对应档口"。

#### 4. 角色级别体系（1-10 级权限控制）
**品智做法**：角色分 10 个级别，高级别可管理低级别。限制折扣上限、抹零上限、赠送上限、数据查询时限。
**屯象OS 现状**：menu_config.py 有角色过滤但只区分 admin/manager/chef/waiter/cashier 5 种，无级别概念。
**借鉴建议**：角色增加 `level (1-10)` 字段 + 折扣上限/抹零上限/赠送上限/查询时限 4 个控制字段。

### 🟡 P1 — 应该借鉴（提升专业度）

#### 5. 支付方式的 13 类别分类 + 实收属性
**品智做法**：支付方式分"闭环类"（自动收款）和"记录类"（人工记录），每种有"计入实收"开关和"实收比例"。
**屯象OS 现状**：PaymentMethod 枚举仅 8 种，无实收比例概念。
**借鉴建议**：Payment 模型增加 `is_actual_revenue: bool` + `actual_revenue_ratio: float` 字段。

#### 6. 营销方案的 7 种类型 + 共享互斥规则
**品智做法**：特价/买赠/加价换购/再买/会员/订单折扣/满减 7 种，有执行顺序和共享互斥规则。
**屯象OS 现状**：DiscountApprovalModal 有折扣审批，但无营销方案配置体系。
**借鉴建议**：tx-member 增加营销方案引擎，支持 7 种方案类型 + 互斥规则。

#### 7. 门店标签的多维度分类
**品智做法**：门店类别（商圈/属性/规模）+ 门店标签（3 类 × N 个标签），可按标签筛选报表。
**屯象OS 现状**：Store 模型有 brand_id/region，无标签体系。
**借鉴建议**：Store 增加 `tags: JSON` + `store_category: String`，支持多维度分析。

#### 8. 支付方式沉睡天数（自动清理不用的配置）
**品智做法**：统计支付方式/优惠券多少天未使用，超过阈值可批量停用/删除。
**屯象OS 现状**：无此概念。
**借鉴建议**：支付方式和营销方案增加 `last_used_at` 字段，定期清理沉睡配置。

### 🟢 P2 — 可以借鉴（锦上添花）

#### 9. 企业主视觉图片（品牌化登录页）
**品智做法**：上传企业logo后，登录页、小票、配送单自动替换品牌标识。
**借鉴建议**：Store/Brand 模型增加 `logo_url` + `receipt_logo_url`，品牌化体验。

#### 10. 钉钉/企微/飞书扫码登录
**品智做法**：支持账号/扫码/混合 3 种登录方式，离职自动停用。
**屯象OS 现状**：Employee 有 wechat_userid/dingtalk_userid 字段但未实现扫码登录。
**借鉴建议**：gateway 增加 OAuth2 扫码登录端点。

#### 11. 自定义列表显示列（用户维度存储）
**品智做法**：门店列表支持自定义显示列、排序、锁定列，设置信息按用户维度存储。
**借鉴建议**：前端增加列表配置持久化（localStorage 或后端 user_preferences）。

#### 12. 菜品售卖日期定时上下架
**品智做法**：菜品可设置"指定日期售卖"+"指定时间段售卖"，自动定时上下架。
**借鉴建议**：Dish 模型增加 `sell_start_date/sell_end_date/sell_time_ranges` 字段。

## 三、屯象OS 已超越品智的能力（品智没有的）

| # | 屯象OS 独有能力 | 品智极智版 |
|---|-------------|--------|
| 1 | **9 大 AI Agent（73 action）** | 无任何 AI 能力 |
| 2 | **三条硬约束自动校验** | 仅有角色级折扣上限（被动） |
| 3 | **经营叙事引擎（≤200字简报）** | 无 |
| 4 | **5 因子菜单排名** | 无 |
| 5 | **损耗 TOP5 + 根因映射** | 无损耗管理模块 |
| 6 | **日清日结 E1-E8 自动化** | 仅有日结/班别设置 |
| 7 | **Mac mini 边缘 AI（断网可用）** | 纯云端，断网停摆 |
| 8 | **Agent Console（实时决策+NL问数+审计）** | 无 |
| 9 | **桌台/订单完整状态机（8+9状态）** | 桌台仅基础状态 |
| 10 | **行业模板 Pro/Standard/Lite** | 单一版本 |
| 11 | **Ontology 全链路溯源** | 数据孤岛 |
| 12 | **4 时间点智能推送** | 无主动推送 |

## 五、一句话总结

> **品智极智版是一个成熟的"工具型"POS 系统，在连锁门店运营的配置深度上值得学习（特别是快速开店、菜品发布、出品部门联动、角色控制这 4 项）。但它本质上是"传统餐饮软件"——没有 AI、没有主动决策、没有全链路溯源。屯象OS 应该借鉴它的运营细节打磨能力，同时保持 AI-Native 的根本差异化。**
